## 답을 본 뒤 알게 된 것

### CORS란?

- 막는 사람: 브라우저
- 풀어주는 사람: 백엔드
- 막힌 사람: 나

- 브라우저는 왜 출처가 다른 요청을 경계할까?
- 내 예상: 정상적인 서비스라면, '자기들끼리' 이야기 하는게 맞지 않을까? 내가 겪은 상황은 localhost가 실제 백엔드 서버와 소통할때 발생했었고, 정상적인 서비스라면 api.blahblah.com과 www.blahblah.com이 소통하는게 당연하다. 외부에서 api 요청을 보냈는데 그걸 받아주면, 오히려 이상하게 느껴지고, 그렇기 때문에 CORS를 잘 몰라도 막히는게 이상하게 느껴지진 않는다.

- 정답 시나리오
  - 정확해. 방향을 제대로 잡았어.
  - 네 직관 그대로야 — 정상적인 서비스라면 "우리 프론트"랑 "우리 백엔드"끼리만 대화하는 게 자연스럽고, 생판 모르는 외부 출처가 우리 API를 찔러대는 건 의심스러운 상황이지.
  - 근데 여기서 CORS가 진짜로 막으려는 시나리오를 딱 하나 보여줄게. 이게 핵심이야.
  - 너가 오전 10시에 은행.com에 로그인했어. 서버가 세션 쿠키를 줬고, 브라우저가 은행.com 소속으로 저장소에 저장함. 그리고 은행 탭을 닫았어. 근데 여기서 - - 중요한 게 — 쿠키는 탭을 닫아도 안 사라져. 만료 시간 전까지 저장소에 계속 살아있어.
  - 오전 10시 20분, 다른 탭에서 실수로 해커.com에 접속했어. 은행 탭은 이미 닫힌 상태고. 근데 해커 사이트의 JS 코드가 이렇게 요청을 날려:

  ```js
  fetch("https://은행.com/송금", { credentials: "include" });
  ```

  - 브라우저 규칙이 이래: 요청 목적지가 은행.com이면, 은행.com 소속 쿠키를 자동으로 붙인다. 이때 "누가 이 요청을 시작했는지"(해커.com 페이지)는 안 따져. - 목적지 기준으로만 판단해. 그래서 은행 서버 입장에선 "어 로그인된 사용자가 송금 요청했네?" 하고 처리해버릴 수 있어. 네가 시키지도 않았는데 해커 사이트가 네 로그인 상태를 훔쳐 쓰는 거지.
  - 여기서 위험 구간은 탭이나 시간이 아니라 쿠키가 저장소에 살아있는 동안이야. 로그인 유지가 7일이면 7일 내내 위험한 거고.
  - 그래서 브라우저가 "출처가 다른 요청"을 기본적으로 경계하는 거야. 네 직관("외부에서 요청하는 건 이상하다")이 정확히 이 위협을 감지한 거고.
  - 질문. 그럼 방금 이 시나리오에서 — 나쁜 요청을 보낸 출처는 해커.com이고, 요청받은 서버는 은행.com이야. 이때 "허용할지 말지"를 결정하는 헤더는 누가 붙여야 할까? 해커야, 은행이야?

- 내 대답: 이거는 프론트에서 손 쓸 수 없던 것으로 기억한다. 그리고 무엇보다 프론트에서 건들 수 있다면 CORS 설정이 의미 없지않나? 백엔드인 서버, 즉 은행이다.

- 은행 서버가 응답에 이 헤더를 붙여.

```
Access-Control-Allow-Origin: https://우리프론트.com
```

- 이건 은행이 "우리 프론트에서 오는 요청만 허용할거야"라고 선언한 거야.

- 너가 한 것도, "너희 서버 응답에 우리 출처를 Allow-Origin에 넣어줘"라고 부탁한거지.

- Q. 왜 'Origin'이야? 이름이 왜 그래?
- A. Origin: https://해커.com 을 요청 헤더에 보내기 때문.

- Q. 그럼 Origin을 변조하면 되잖아.
- A. 브라우저 안에서는 변조 불가, 그러나 브라우저 밖- 즉, postman, curl, 직접 짠 코드에서는 변조 가능. 그러나 의미 없음. CORS는 애초에 브라우저를 쓰는 일반 사용자를 보호하기 위한 장치. 현재 상황에서는 은행에 로그인해뒀지만 실수로 해커 사이트에 들어간 사람. 밖에서의 위조는 위협이 안 된다.

- 역질문 Q. CORS 차단은 브라우저가 응답을 이미 받은 뒤에 "이 출처 허용 안 됐네" 하고 프론트 코드에 안 넘기는 방식이라고 했잖아.
  그럼 이상하지 않아? 아까 그 해커 시나리오 — fetch('https://은행.com/송금') 보낸 거. 브라우저가 응답을 받았다는 건, 은행 서버에 송금 요청이 이미 도착해서 처리됐다는 뜻이잖아. 차단은 응답 단계에서 일어나는데, 송금은 이미 서버에서 실행돼버린 거 아니야?
  그럼 CORS가 이 공격을 실제로 막긴 막은 거야? 어디를 막은 거지?
  (힌트: "막았다"와 "못 막았다"가 둘 다 부분적으로 맞아. 뭘 막고 뭘 못 막았는지 나눠서 생각해봐.)

- A. 모르겠음...

- 정답: 못 막은 것: 요청 자체는 서버에 도착했다.
  브라우저는 요청을 실제로 보내. 그래서 단순 요청(Simple request)이면 은행 서버에 /송금이 도착하고, 서버가 그걸 처리해버릴 수 있어. CORS는 이 요청이 나가는 걸 못 막아. 여기가 네 직관("송금 이미 실행된 거 아냐?")이 맞은 부분이야.
  막은 것: 응답을 못 읽게 했다.
  브라우저는 응답을 받지만, 출처가 허용 안 됐으면 그 응답 내용을 해커.com의 JS한테 안 넘겨. 그래서 해커는 송금 결과(잔액, 응답 데이터)를 못 읽어.
  즉 CORS가 진짜 지키는 건 **"응답 데이터 읽기"**야. "요청 보내기"가 아니라.
  그래서 이런 결론이 나와:
  데이터를 읽어가는 공격(예: 은행 잔액 조회해서 빼가기) → CORS가 막아. 응답을 못 읽으니까.
  서버 상태를 바꿔버리는 공격(예: 송금 실행) → CORS만으론 못 막아. 요청은 이미 도착했으니까.
  그래서 네가 떠올린 트랜잭션/멱등성 쪽 감이 왜 나왔냐면 — "상태를 바꾸는 요청"을 막는 건 CORS 담당이 아니라 CSRF 토큰 같은 별도 방어의 몫이거든. CORS랑 CSRF 방어가 역할이 갈리는 지점이 정확히 여기야.
  그리고 바로 이 "상태 바꾸는 요청은 위험하다"는 문제의식 때문에 — 브라우저가 위험해 보이는 요청은 아예 보내기 전에 서버한테 먼저 물어보는 장치를 만들었어. 그게 Preflight야.

- 이제 다음 단계로 가면 — 방금 "브라우저가 응답을 받고 나서 차단한다"고 했잖아? 근데 어떤 요청은 브라우저가 아예 보내기 전에 서버한테 "이거 보내도 돼?"라고 미리 물어봐. 이게 **Preflight(사전 요청)**이야. 면접 단골이고.

### Preflight(사전 요청)

- 자 이제 Preflight. 방금 흐름에서 자연스럽게 나왔어 — "상태 바꾸는 요청은 위험하다"는 문제의식.
  브라우저가 요청을 두 종류로 나눠:
  Simple request(단순 요청) — 그냥 보냄. 응답 받고 나서 CORS 검사. (지금까지 얘기한 게 이거)
  Preflight가 붙는 요청 — 위험해 보이는 요청은, 진짜 요청을 보내기 전에 브라우저가 먼저 "이거 보내도 돼?"라고 서버한테 물어보는 예비 요청을 하나 날려. 이게 Preflight야. 서버가 "응 돼"라고 허락하면 그때 진짜 요청을 보내고, "안 돼"면 진짜 요청은 아예 안 나가.

- Q. 그러면? HTTP 메서드 중에 어떤게 Preflight를 유발할 것 같아?
- A. 멱등성은 결과가 같느냐의 이야기, 즉 사용자의 관점이라고 선을 그을 것 같아. 그러면 CRUD로 접근하는게 맞는 것 같고. 그러면 C,U,D. 즉, POST, PATCH, PUT, DELETE가 맞는 듯.

- 정답: 거의 정확해. 그리고 "멱등성은 사용자 관점, 여기선 서버 영향"이라고 선 그은 거 — 그거 아주 좋은 분리야. 실제로 그게 맞아.
  근데 딱 하나, POST는 예외적으로 Simple request가 될 수 있어. 여기가 함정이야.
  CORS 스펙에서 "Simple request(그냥 보내도 되는 요청)"으로 쳐주는 메서드는 딱 3개야: GET, HEAD, POST.
  어? POST가 왜 여기 껴? 상태 바꾸는데?
  이유가 이거야 — CORS는 새로 생긴 규칙인데, POST로 폼 전송하는 건 CORS 나오기 훨씬 전부터 웹에 원래 있던 기능이거든. HTML <form> 태그로 POST 보내는 거. 이건 CORS가 생기기 전부터 브라우저가 하던 거라, 이걸 갑자기 Preflight로 막아버리면 기존 웹이 다 깨져. 그래서 하위 호환 때문에 POST를 Simple 목록에 남겨둔 거야.
  그래서 네 답을 교정하면:
- PUT, PATCH, DELETE → 무조건 Preflight 유발. (네 말 맞음)
  POST → 메서드만 보면 Simple. 근데 다른 조건 때문에 Preflight로 넘어가는 경우가 대부분이야.
- 그 "다른 조건"이 핵심인데 — 아까 우리가 CORS 시작할 때 잠깐 스쳤던 거야. Content-Type: application/json. 실무에서 POST로 JSON 보내면 Preflight 뜨는 이유가 이거야.
  - Q. 왜 Content-Type: application/json이 Preflight를 유발할까? 힌트: 방금 말한 "원래 HTML form이 하던 것"과 관련 있어. HTML form이 보낼 수 있는 Content-Type이 정해져 있거든. 추측해볼래? 아니면 바로 풀어줄까?

  - A. HTML <form> 태그는 CORS 생기기 전부터 있던 기능인데, 얘가 보낼 수 있는 Content-Type이 딱 3개로 정해져 있어:
    application/x-www-form-urlencoded (기본값, key=value&key=value 형식)
    multipart/form-data (파일 업로드용)
    text/plain (그냥 텍스트)

    이 3개는 **"옛날부터 HTML form이 보내던 거"**라서 Simple request로 쳐줘. 하위 호환 때문에.

    근데 application/json은? 이 목록에 없어. JSON은 HTML form이 원래 보내던 형식이 아니거든. JS로 fetch 써서 Content-Type: application/json으로 보내는 건 "옛날 웹에 없던 새로운 행위"야. 그래서 브라우저가 "어 이건 form이 하던 게 아니네? 위험할 수 있으니 서버한테 먼저 물어보자" 하고 Preflight를 띄워.
    그래서 실무에서 이렇게 되는 거야:

```js
fetch("https://api.com/users", {
  method: "POST",
  headers: { "Content-Type": "application/json" }, // ← 이것 때문에 Preflight
  body: JSON.stringify({ name: "hyun" }),
});
```

- 정리하면 Preflight 유발 조건은 크게:

1. 메서드가 GET/HEAD/POST 밖 → PUT, PATCH, DELETE 등
2. Content-Type이 저 3개 밖 → application/json 등
3. 커스텀 헤더 붙임 → 예: Authorization, X-Custom-...

### Preflight는 어떻게 생겼을까?

- Preflight는 OPTIONS 메서드로 나가. 브라우저가 진짜 요청 보내기 전에 자동으로 하나 먼저 쏘는 거야. 아까 그 JSON POST 예시로 보자:

```js
fetch("https://api.com/users", {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({ name: "hyun" }),
});
```

1. 1단계, Preflight 요청

```
OPTIONS /users HTTP/1.1
Origin: https://우리프론트.com
Access-Control-Request-Method: POST
Access-Control-Request-Headers: Content-Type
```

- "나 우리 프론튼데, POST 메서드로, Content-Type 붙여서, /user 보낼거임. 됨?" -> 바디는 없는 요청

2. 서버 응답

```
HTTP/1.1 204 No Content
Access-Control-Allow-Origin: https://우리프론트.com
Access-Control-Allow-Methods: POST, GET, DELETE
Access-Control-Allow-Headers: Content-Type
```

- "ㅇㅇ 우리프론트 괜찮고 POST/GET/DELETE면 OK. Content-Type 헤더도 허용함"

3. 브라우저의 대조

```
브라우저 → OPTIONS(물어봄) → 서버
브라우저 ← 204(허락함) ← 서버
브라우저 → POST(진짜 요청) → 서버   ← 여기서 처음으로 body 감
브라우저 ← 200(진짜 응답) ← 서버
```

- 아까 배운 거랑 연결되는 포인트 하나. Preflight가 붙는 요청은 "요청 보내기" 자체도 막을 수 있어. 서버가 OPTIONS에서 거절하면 진짜 요청이 아예 안 나가거든. 그래서 아까 "CORS는 요청 보내기를 못 막는다"고 한 건 Simple request 얘기였고, Preflight 붙는 요청은 진짜 요청 전에 걸러지니까 상태 변경도 사전 차단이 돼. 이게 왜 위험한 메서드/헤더에 Preflight를 붙이는지의 이유야.

- 실무 팁 하나: Preflight를 매번 하면 요청이 2배가 되니까, 서버가 Access-Control-Max-Age: 86400 이런 헤더로 "이 허락 하루 동안 캐싱해"라고 하면 브라우저가 그 시간 동안 OPTIONS를 생략해.

- Q. GET 요청은 Preflight 뜰까?
- A. GET은 Body가 없지 않을까? 안된다. 심플 리퀘스트.

- 정답: GET도 body를 담는 게 기술적으로 불가능하지 않음. 진짜 이유는, 메서드가 simple 목록에 들어있기 때문이니까(GET/HEAD/POST).

- 그런데 함정, GET도 Preflight 뜰 수 있다.

- GET도 Authorization같은 커스텀 헤더가 붙으면(Bearer XX) Preflight 뜸. GET이니까 안 뜬다가 아니라, 조건 3개가 걸려야 함

1. 메서드가 GET/HEAD/POST 밖이냐? → GET은 목록 안. 통과.
2. Content-Type이 그 3개 밖이냐? → GET은 보통 body가 없으니 Content-Type 자체가 문제 안 됨. 통과.
3. 커스텀 헤더 붙었냐? → 안 붙었으면 통과.

```
1. 삼각구도
막는 놈 = 브라우저 / 푸는 놈 = 서버 / 낀 놈 = 프론트. 네가 예전에 백엔드 찾아간 게 정답이었던 이유 = 허용은 서버만 정할 수 있으니까.
2. 왜 존재하나 (SOP)
브라우저가 다른 출처 요청을 경계하는 건, 로그인 쿠키를 악용한 공격(은행 로그인 상태로 해커 사이트 접속 시나리오)을 막기 위해서. 쿠키는 목적지 도메인 기준으로 자동 첨부되고, 저장소에 살아있는 동안 위험 구간.
3. 출처(Origin)의 정의
scheme + host + port 세 개 조합. 하나라도 다르면 다른 출처.
4. 헤더 대조 구조 (핵심)

요청 헤더: Origin: ... — 브라우저가 강제로 박음. JS 위조 불가(Forbidden header).
응답 헤더: Access-Control-Allow-Origin: ... — 서버가 허용 출처 선언.
브라우저가 둘 대조 → 안 맞으면 응답을 프론트에 안 넘김.

5. CORS가 진짜 지키는 것
"요청 보내기"가 아니라 **"응답 읽기"**를 막는 거. → 데이터 탈취는 막고, 상태 변경(송금)은 CORS만으론 못 막음 → 그건 CSRF 방어 몫.
6. Preflight
위험한 요청은 진짜 요청 전에 OPTIONS로 "보내도 돼?" 먼저 물어봄. 서버가 Allow-Methods/Allow-Headers로 허락하면 그때 진짜 요청 감. (Preflight 붙는 요청은 요청 자체도 사전 차단 가능.)
7. Preflight 발동 조건 3개 (판정 기준)

메서드가 GET/HEAD/POST 밖 (PUT/PATCH/DELETE)
Content-Type이 x-www-form-urlencoded/multipart/form-data/text/plain 밖 (→ application/json)
커스텀 헤더 (Authorization, X-...)

→ 셋 중 하나라도 걸리면 Preflight. 메서드 단독이 아니라 조건 3개로 판정.
8. 면접 킬포인트 3개

"Postman/curl은 왜 CORS 안 걸림?" → CORS는 브라우저 유저 보호 장치. 브라우저 밖은 대상 아님.
"CORS가 뭘 지킴?" → 응답 읽기. 요청 보내기 아님.
"왜 JSON POST는 Preflight 뜸?" → Content-Type이 HTML form 3종 밖이라서.
```

## 틀렸거나 부족했던 부분

Q. 쿠키는 언제 자동으로 붙어?
A. 요청 목적지 도메인 기준. 요청을 누가 시작했는지가 아니라, 어디로 가는지로 판단해서 그 도메인 소속 쿠키를 자동 첨부.

Q. 쿠키 위험 구간은 탭 열려있는 동안이야?
A. 아니. 탭/시간 무관. 쿠키가 저장소에 살아있는(만료 전) 동안이 위험 구간.

Q. 요청에도 출처 헤더가 붙어?
A. 응. 브라우저가 요청에 Origin: ...을 강제로 박아. (응답의 Access-Control-Allow-Origin만 있는 게 아님.)

Q. Origin 변조 되나?
A. 브라우저 안에선 불가(Forbidden header, JS가 못 건드림). 밖(curl/Postman)에선 가능하지만, CORS는 브라우저 유저 보호가 목적이라 위협 안 됨.

Q. CORS는 뭘 막는 거야?
A. "응답 읽기"를 막는 거지 "요청 보내기"를 막는 게 아님. → 데이터 탈취는 막고, 상태 변경(송금)은 CSRF 방어 몫.

Q. POST는 상태 바꾸는데 왜 Simple이야?
A. HTML form이 CORS 이전부터 POST를 쓰던 하위 호환 때문. Simple 메서드는 GET/HEAD/POST 3개.

Q. 그럼 JSON POST는 왜 Preflight 떠?
A. Content-Type: application/json이 HTML form 3종(x-www-form-urlencoded, multipart/form-data, text/plain) 밖이라서.

Q. GET은 무조건 Preflight 안 떠?
A. 아니. 커스텀 헤더(예: Authorization) 붙으면 GET도 Preflight. 판정은 메서드 단독이 아니라 조건 3개로.

Q. Preflight 매번 뜨면 요청 2배 아냐?
A. Access-Control-Max-Age로 허락을 캐싱하면 그 시간 동안 OPTIONS 생략.

## 면접용 1분 답변

- "CORS가 정확히 뭘 막는 건지 설명해주시고, Preflight가 왜 필요한지도 같이 말씀해주세요."

- CORS는 크로스 오리진 리소스 쉐어링입니다. 허용하지 않은 출처로부터의 요청은 막지 않지만 응답을 막습니다. 허용한 출처로 보내는 위장 요청에 대해 일반 사용자를 보호하기 위해 존재하며, 요청을 막을 수는 없기에 Preflight가 존재합니다. Preflight는 주로 GET HEAD POST에서 사용되며, 컨텐츠 타입이 뭔지, 그리고 커스텀 헤더가 있는지를 판단합니다.

(연계 개념: CSRF, SOP, 쿠키 보안 속성, 세션, JWT)
