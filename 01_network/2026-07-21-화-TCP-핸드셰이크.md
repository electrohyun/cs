## 답을 본 뒤 알게 된 것

### TCP란?

- Transmission Control Protocol 전송 연결 프로토콜 (연결형 전송 프로토콜)
- UDP란? User Datagram Protocol

- IP가 보장하지 않는 것? -> 패킷 유실, 순서, 중복, 목적지까지 보내려는 시도만

- TCP의 선호

1. 연결 지향 — 데이터 전송 전에 3-way handshake로 양쪽 상태를 맞춤. 종료는 4-way.
2. 신뢰성 — 받은 쪽이 ACK를 보내고, 보낸 쪽은 타임아웃 안에 ACK가 안 오면 재전송. 중복 ACK가 쌓이면 빠른 재전송.
3. 순서 보장 — 모든 바이트에 시퀀스 넘버가 붙음. 수신 측이 버퍼에 담아 순서 맞춰서 애플리케이션에 전달.
4. 흐름 제어 — 수신 측이 헤더의 윈도우 크기 필드로 "지금 이만큼 받을 수 있다"를 알림. 송신 측은 그 이상 못 보냄. 수신 버퍼 오버플로 방지.
5. 혼잡 제어 — 네트워크 상황에 따라 송신량 조절. Slow Start, Congestion Avoidance, Fast Recovery.

- 바이트 스트림
- TCP는 데이터를 메시지 단위가 아니라 경계 없는 바이트 열로 취급.
- send() 두 번이 수신 측에서 한 번에 읽힐 수도, 쪼개져 읽힐 수도 있음.
- 그래서 상위 프로토콜이 경계를 직접 정의함 — HTTP의 Content-Length, WebSocket의 프레임 헤더가 그 역할.

"인터넷 연결이 반드시 TCP/UDP인 건 아니지만, 애플리케이션이 데이터를 주고받는 통신은 사실상 거의 다 TCP 아니면 UDP임. 웹, 메일, 파일 전송, 게임, 스트리밍 전부. 그리고 "연결"이라는 말이 붙을 수 있는 건 여전히 TCP 쪽. UDP나 IP는 연결 상태가 없음." - Claude

핵심 차이는 **신뢰성을 보장하냐 vs 지연을 줄이냐**의 트레이드오프.

|             | TCP                           | UDP                            |
| ----------- | ----------------------------- | ------------------------------ |
| 연결        | 3-way handshake로 수립        | 없음, 바로 전송                |
| 상태        | 있음 (LISTEN, ESTABLISHED 등) | 없음                           |
| 유실 처리   | ACK + 타임아웃 재전송         | 없음, 그대로 손실              |
| 순서        | 시퀀스 넘버로 보장            | 없음, 도착 순서대로            |
| 흐름 제어   | 수신 윈도우                   | 없음                           |
| 혼잡 제어   | Slow Start 등                 | 없음                           |
| 데이터 단위 | 바이트 스트림 (경계 없음)     | 데이터그램 (경계 유지)         |
| 헤더        | 최소 20바이트                 | 8바이트                        |
| 1:N 전송    | 불가 (1:1만)                  | 가능 (브로드캐스트/멀티캐스트) |

**동영상에서 UDP가 유리한 진짜 이유**

재전송이 무의미해서만이 아님. TCP는 순서 보장 때문에 **앞 패킷 하나가 유실되면 뒤에 이미 도착한 패킷들도 애플리케이션에 못 올려줌.** 버퍼에 쌓아두고 빠진 게 재전송돼 도착할 때까지 대기함. 이걸 Head-of-Line Blocking이라고 함.

실시간 영상에서는 프레임 하나 깨지는 것보다 이 대기로 인한 끊김이 훨씬 치명적임. UDP는 빠진 걸 건너뛰고 그냥 다음 걸 재생함.

참고로 HTTP/2가 TCP 위에서 여러 스트림을 다중화했는데도 이 문제를 못 벗어나서, HTTP/3가 UDP 기반 QUIC으로 간 이유가 이거임.

**주의할 오해**

UDP가 "항상 빠른" 건 아님. 헤더가 작고 handshake가 없어서 초기 지연이 적을 뿐, 대역폭이 넉넉하면 전송 속도 자체는 큰 차이 안 남. 오히려 혼잡 제어가 없어서 네트워크가 막힐 때 무작정 계속 보내다 유실률이 폭증할 수 있음.

그리고 UDP 쓰는 서비스들도 신뢰성이 필요하면 애플리케이션 레벨에서 직접 구현함. QUIC이 그 극단적인 예.

### 3-way Handshake

- TCP(Transmission Control Protocol, 신뢰성 있는 연결형 전송 프로토콜)가 데이터를 주고받기 전에, 양쪽이 서로의 시작 번호를 교환하고 양방향 통신이 가능한지 확인하는 3단계 절차. 패킷 3개가 오가서 3-way(3번 왕복이 아니라 3개 메시지)임.

- 준비: 서버는 listen()을 호출해 LISTEN(들어오는 연결 요청을 기다리는 상태)으로 대기 중.

1. SYN (클 -> 서)

- SYN(Synchronize, 동기화 요청 플래그) = 1
- 시퀀스 넘버 자리에 ISN(c)(Initial Sequence Number of client, 클라이언트가 정한 랜덤 시작 번호)를 담음
- 클라이언트 상태: SYN_SENT(요청 보내고 응답 기다리는 상태)

- 의미: "연결하고 싶다. 내 번호는 이거부터 시작한다."

2. SYN + ACK

- SYN = 1, ACK(Acknowledgment, 확인 응답 플래그) = 1
- 시퀀스 넘버 자리에 ISN(s)(서버가 정한 랜덤 시작 번호)
- ACK 넘버(다음에 받기를 기대하는 번호) = ISN(c) + 1
- 서버 상태: SYN_RECEIVED(요청 받고 마지막 확인을 기다리는 상태)

- 의미: "네 요청 받았다(ACK). 나도 연결하겠다, 내 번호는 이거다(SYN)."
- 서버 쪽 SYN과 ACK를 따로 보내도 되지만 한 패킷에 합쳐서 보냄. 이게 4번이 아니라 3번인 이유.

3. ACK

- ACK = 1
- ACK 넘버 = ISN(s) + 1
- 클라이언트 상태: ESTABLISHED(연결 수립 완료, 데이터 송수신 가능)
- 서버도 이 패킷을 받으면 ESTABLISHED

- 의미: "네 번호도 받았다. 시작하자."

#### 왜 +1인가, 왜 3번인가

- SYN 플래그는 데이터가 없어도 시퀀스 넘버를 1 소비함. 그래서 "다음에 기대하는 번호"가 ISN+1이 됨. FIN(Finish, 종료 요청 플래그)도 마찬가지.

- 각 방향마다 "내 번호 알림(SYN)"과 "받았다는 확인(ACK)"이 필요하니 논리적으론 4개. 그런데 서버 쪽 두 개를 한 패킷에 합칠 수 있어서 3개로 줄어듦.
  2개로는 안 됨. 2단계까지만 하면 서버는 클라이언트가 자기 SYN을 받았는지 모름. 클라이언트→서버 방향만 검증되고 반대 방향은 미검증 상태로 남음.

#### SYN Flood

**공격 원리**

서버는 2단계(SYN+ACK)를 보낸 시점에 아직 연결이 완성 안 됐지만, 그 연결 정보를 어딘가 저장해둬야 함. 3단계 ACK가 왔을 때 "아까 그 요청"과 맞춰야 하니까.

이 저장소가 **백로그 큐**(backlog queue, SYN_RECEIVED 상태의 미완성 연결을 담아두는 대기열). 크기가 유한함.

공격자는 SYN만 계속 보내고 3단계 ACK를 절대 안 보냄. 게다가 출발지 IP를 위조하면 SYN+ACK이 엉뚱한 곳으로 가서 응답이 아예 안 옴.

결과: 큐가 반쯤 열린 연결(**half-open connection**)로 가득 참 → 정상 사용자의 SYN이 들어와도 자리가 없어서 거부됨. **DoS**(Denial of Service, 서비스 거부).

**왜 효과적인가**

공격자는 SYN 패킷 하나(수십 바이트) 던지면 끝. 서버는 연결 정보 저장 + 타이머 설정 + SYN+ACK 재전송(보통 여러 번, 수십 초간)까지 함. **비용 비대칭**이 커서 적은 자원으로 큰 부담을 줄 수 있음.

#### SYN Cookie

**핵심 아이디어: 저장하지 않는다.**

큐가 찼을 때, 서버는 연결 정보를 메모리에 저장하는 대신 **그 정보를 인코딩해서 ISN(s) 안에 넣어 보냄.**

ISN(s) 32비트를 이렇게 씀:

```
[상위 5비트] 타임스탬프 카운터
[다음 3비트] MSS 인덱스 (Maximum Segment Size, 한 세그먼트 최대 크기)
[하위 24비트] 해시값
```

해시값 = `H(출발지IP, 출발지포트, 목적지IP, 목적지포트, 타임스탬프, 서버 시크릿키)`

**동작**

1. SYN 받음 → 위 공식으로 ISN(s) 계산해서 SYN+ACK 전송. **아무것도 저장 안 함.**
2. 3단계 ACK 도착 → ACK 넘버에서 1 빼면 서버가 보냈던 ISN(s)가 나옴
3. 그 패킷의 IP/포트로 해시를 다시 계산해서 일치하는지 검증
4. 일치하면 정상 요청이므로 그때 연결 구조체 생성

저장을 안 하니 큐가 넘칠 일이 없음. 공격자가 SYN을 아무리 보내도 서버 메모리는 안 늘어남.

시크릿키를 모르면 유효한 값을 위조할 수 없고, 타임스탬프가 들어가서 오래된 쿠키는 만료됨.

**단점**

- ISN에 24비트만 남아서 TCP 옵션(윈도우 스케일링, SACK 등)을 온전히 담을 수 없음 → 성능 저하
- 그래서 리눅스는 평소엔 끄고, **큐가 실제로 넘칠 때만 자동 발동**함 (`net.ipv4.tcp_syncookies=1`)

**다른 방어책**

- 백로그 큐 크기 확대
- SYN+ACK 재전송 횟수 축소 (`tcp_synack_retries`)
- 방화벽/로드밸런서가 대신 핸드셰이크 완료 후 뒷단에 전달 (SYN Proxy)
- 클라우드 DDoS 방어 서비스

---

정리하면 3-way handshake의 구조적 약점 = "서버가 절반만 완성된 상태를 기억해야 함". SYN Cookie는 그 기억을 클라이언트에게 떠넘기는 해법이고, 상시로 켜놓지는 않는 비상 스위치이다.

### 4-way handshake (연결 종료)

- 연결은 3번인데 종료는 4번임. 연결은 대칭이지만 종료는 비대칭이기 때문.이유: TCP는 양방향 스트림이라 각 방향을 따로 닫음. A가 "할 말 끝났다"고 해도 B는 아직 보낼 게 남아있을 수 있음. 그래서 SYN처럼 합칠 수가 없음.

1. FIN (클 -> 서)

- FIN(Finish, 더 보낼 데이터 없음을 알리는 플래그) = 1
- 클라이언트: FIN_WAIT_1

- 의미: "내 방향은 닫는다. 근데 받는 건 계속 받을 수 있음."

2. ACK (서 -> 클)

- ACK 넘버 = FIN의 시퀀스 넘버 + 1
- 서버: CLOSE_WAIT(상대는 닫았고, 내 쪽 종료를 준비하는 상태)
- 클라이언트: FIN_WAIT_2

- 이 시점부터 half-close(반만 닫힌 상태). 서버 → 클라이언트 방향은 아직 살아있어서 데이터 전송 가능.

3. FIN (서 -> 클)

- 서버 애플리케이션이 close() 호출하면 그때 나감
- 서버: LAST_ACK

4. ACK (클 -> 서)

- 서버: CLOSED
- 클라이언트: TIME_WAIT → 일정 시간 후 CLOSED

#### 2번과 3번이 왜 안 합쳐지나

- 2단계 ACK는 커널이 즉시 보냄. 3단계 FIN은 애플리케이션이 close()를 호출해야 나감. 그 사이에 서버가 남은 데이터를 보낼 수 있어서, 시점이 다름.
  보낼 게 없으면 합쳐지는 경우도 실제로 있음(3-way로 끝나는 종료). 하지만 표준 시나리오는 4단계.

#### CLOSE_WAIT이 쌓이면

- 서버에 CLOSE_WAIT 소켓이 대량으로 남아있다면 애플리케이션이 close()를 안 부르고 있다는 신호. 커널 문제가 아니라 코드 버그임. 커넥션 반납 누락 같은 것.
  실무에서 netstat으로 CLOSE_WAIT 많이 보이면 백엔드 코드부터 의심하는 이유가 이거.

#### TIME_WAIT

- 마지막 ACK를 보낸 쪽(먼저 닫은 쪽)이 바로 CLOSED로 안 가고 2MSL(Maximum Segment Lifetime의 2배, 보통 30초~2분) 동안 머무는 상태.
  MSL은 패킷이 네트워크에서 살아있을 수 있는 최대 시간.
  왜 필요한가 — 이유 2개

1. 마지막 ACK가 유실될 수 있음
   4단계 ACK가 사라지면 서버는 LAST_ACK에서 FIN을 재전송함. 클라이언트가 이미 CLOSED면 그 FIN에 RST(연결 거부)로 답해서 서버가 정상 종료를 못 함. TIME_WAIT으로 남아있어야 재전송된 FIN에 다시 ACK를 줄 수 있음.
2. 옛날 패킷이 새 연결에 섞이는 것 방지
   같은 IP:포트 쌍으로 바로 재연결하면, 지연됐던 이전 연결 패킷이 새 연결로 들어올 수 있음. 2MSL을 기다리면 그런 패킷은 전부 수명이 다해 사라짐. (ISN 랜덤화와 같은 문제를 다른 층에서 막는 것)
   왜 2배인가
   마지막 ACK가 가는 시간(최대 1MSL) + 상대의 FIN 재전송이 오는 시간(최대 1MSL).

#### TIME_WAIT이 실무에서 문제 되는 지점

- TIME_WAIT은 포트를 점유함. 짧은 연결을 대량으로 만드는 서버(예: 매 요청마다 백엔드에 새 연결)는 포트가 고갈될 수 있음.

- SO_REUSEADDR — 서버 재시작 시 "이미 사용 중인 주소" 에러 회피
- tcp_tw_reuse — 클라이언트 측에서 TIME_WAIT 소켓 재사용 허용
- Keep-Alive / 커넥션 풀 — 근본 해법. 연결을 끊지 않고 재사용

- 중요 포인트: TIME_WAIT은 먼저 close()를 호출한 쪽에 생김. HTTP에서 서버가 먼저 끊는 구조면 서버에 쌓이고, 클라이언트가 끊으면 클라이언트에 쌓임.

### TCP 상태 전이 전체 정리

#### TCP 상태 전이 전체

**클라이언트(능동 개방/능동 종료)**

```
CLOSED
  ↓ connect() → SYN 전송
SYN_SENT
  ↓ SYN+ACK 수신 → ACK 전송
ESTABLISHED
  ↓ close() → FIN 전송
FIN_WAIT_1
  ↓ ACK 수신
FIN_WAIT_2
  ↓ FIN 수신 → ACK 전송
TIME_WAIT
  ↓ 2MSL 경과
CLOSED
```

**서버(수동 개방/수동 종료)**

```
CLOSED
  ↓ listen()
LISTEN
  ↓ SYN 수신 → SYN+ACK 전송
SYN_RECEIVED
  ↓ ACK 수신
ESTABLISHED
  ↓ FIN 수신 → ACK 전송
CLOSE_WAIT
  ↓ close() → FIN 전송
LAST_ACK
  ↓ ACK 수신
CLOSED
```

#### 상태별 한 줄 요약

| 상태         | 의미                          | 쌓이면                |
| ------------ | ----------------------------- | --------------------- |
| LISTEN       | 연결 요청 대기                | 정상                  |
| SYN_SENT     | SYN 보내고 응답 대기          | 방화벽/서버 다운 의심 |
| SYN_RECEIVED | SYN+ACK 보내고 ACK 대기       | SYN Flood 의심        |
| ESTABLISHED  | 연결 완료, 송수신 가능        | 정상                  |
| FIN_WAIT_1   | FIN 보내고 ACK 대기           | 상대 무응답           |
| FIN_WAIT_2   | 내 방향만 닫힘, 상대 FIN 대기 | 상대가 close() 안 함  |
| CLOSE_WAIT   | 상대 닫음, 내 close() 대기    | **애플리케이션 버그** |
| LAST_ACK     | 내 FIN 보내고 마지막 ACK 대기 | 드묾                  |
| TIME_WAIT    | 2MSL 대기                     | 짧은 연결 대량 발생   |
| CLOSED       | 연결 없음                     | —                     |

#### 핵심 대칭 구조

- SYN_SENT ↔ SYN_RECEIVED (개방 시 양쪽)
- FIN_WAIT_2 ↔ CLOSE_WAIT (반만 닫힌 상태의 양쪽 시점)
- TIME_WAIT ↔ LAST_ACK (종료 마무리)

**동시 개방/동시 종료**

둘 다 동시에 SYN을 보내면 SYN_SENT → SYN_RECEIVED → ESTABLISHED로 가는 경로가 있고, 동시에 FIN을 보내면 CLOSING 상태를 거쳐 TIME_WAIT으로 감. 면접에서 거의 안 나오지만 상태도에는 존재함.

## 면접용 1분 답변

> TCP는 연결형 프로토콜이라, 데이터를 보내기 전에 3-way handshake로 연결을 수립합니다.
>
> 먼저 클라이언트가 SYN 플래그를 켜고 자신의 초기 시퀀스 넘버 ISN(c)를 담아 보냅니다. 서버는 SYN과 ACK를 함께 보내는데, ACK 넘버로 ISN(c)+1을, 시퀀스 넘버로 자신의 ISN(s)를 담습니다. 마지막으로 클라이언트가 ISN(s)+1로 ACK를 보내면 양쪽 모두 ESTABLISHED 상태가 되어 데이터를 주고받을 수 있습니다.
>
> 3번인 이유는, 각 방향마다 시퀀스 넘버 통지와 그에 대한 확인이 필요해 논리적으로는 4단계지만 서버 쪽 SYN과 ACK를 한 패킷에 합칠 수 있기 때문입니다. 2단계로 줄이면 서버가 자신의 SYN을 클라이언트가 받았는지 확인할 수 없어서, 서버→클라이언트 방향이 검증되지 않습니다.
>
> ISN을 랜덤하게 정하는 이유는 두 가지입니다. 이전 연결의 지연된 패킷이 새 연결에 섞이는 것을 막고, 시퀀스 넘버가 예측 가능하면 IP를 위조한 공격자가 연결에 패킷을 주입할 수 있기 때문입니다.
>
> 이 구조의 약점은 서버가 SYN_RECEIVED 상태를 백로그 큐에 저장해야 한다는 점이고, 이를 노린 것이 SYN Flood 공격입니다. 방어책으로는 연결 정보를 저장하지 않고 ISN에 인코딩해 보내는 SYN Cookie가 있습니다.

## 한 줄 요약

- **3-way handshake** — 데이터 전송 전 양쪽이 서로의 초기 시퀀스 넘버를 교환하고 양방향 통신 가능함을 확인하는 절차
- **SYN → SYN+ACK → ACK** — 각 단계는 상대 번호에 +1한 ACK 넘버로 수신을 확인
- **왜 3번** — 논리적으로 4단계지만 서버 쪽 SYN과 ACK를 한 패킷에 합칠 수 있어서
- **왜 2번은 안 되나** — 서버→클라이언트 방향이 검증되지 않음
- **왜 +1** — SYN/FIN은 데이터가 없어도 시퀀스 넘버를 1 소비함
- **ISN 랜덤화** — 이전 연결 패킷 혼입 방지 + 시퀀스 넘버 예측 공격 방지
- **SYN Flood** — ACK를 보내지 않아 백로그 큐를 반열림 연결로 채우는 DoS
- **SYN Cookie** — 연결 정보를 저장하지 않고 ISN에 해시로 인코딩, 큐가 넘칠 때만 발동
