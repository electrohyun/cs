# 브라우저에서 mock한 트래픽은 언제 개발자도구 네트워크 탭에 뜨는가

## 잘 알고 있는지

- 잘 몰랐고, 개발자대회 출품(socket.io mock 라이브러리)을 준비하다가 알게 됐다. "MSW는 서버 없이도 네트워크 탭에 뜨는데 왜 socket.io mock은 안 되지?"라는 질문에서 시작했다. 그 과정에서 알게 된 것들을 사실 위주로 적는다.

## 내가 아는 만큼 설명

- 처음엔 "네트워크 탭 = 서버까지 다녀온 요청을 보여주는 곳"이라고 생각했다.
- 그래서 "mock은 서버를 안 거치니까 네트워크 탭에 안 뜨는 게 당연하다"고만 이해했다.
- MSW가 서버 없이도 네트워크 탭에 뜨는 건 "뭔가 특별한 트릭"이라고 막연하게 생각했고, 그 원리는 설명하지 못했다.

## 답을 본 뒤 알게 된 것

**1. 네트워크 탭이 보는 건 "서버 도달"이 아니라 "네트워크 계층 통과"다.**

- 개발자도구 네트워크 탭은 "요청이 서버까지 갔는가"를 보여주는 게 아니라, "브라우저의 네트워크 계층을 통과했는가"를 보여준다.
- 그래서 서버까지 안 가도, 네트워크 계층에 진입했다면 항목이 뜬다.

**2. MSW(HTTP)가 서버 없이 뜨는 이유 = Service Worker.**

- 앱이 진짜 `fetch()`를 호출한다 → 요청이 네트워크 계층에 진입한다(이 시점에 이미 브라우저가 인지).
- Service Worker가 그 요청을 `fetch` 이벤트로 가로채고, `respondWith()`로 가짜 응답을 돌려준다.
- 서버엔 안 갔지만 "계층 진입"은 실제로 일어났으므로 네트워크 탭에 뜬다(`from ServiceWorker` 표시가 붙는다).
- 즉 MSW의 핵심은 "가짜 응답"이 아니라 "앱이 진짜 네트워크 API를 쓰게 두고, 계층 진입 후에 가로채는 것"이다.

**3. 객체 흉내(mock-socket 등)가 안 뜨는 이유.**

- 앱이 진짜 `WebSocket`/`fetch`를 부르는 게 아니라, 가짜 JS 객체의 메서드를 부른다.
- 데이터가 브라우저 메모리 안 객체끼리 함수 호출로만 오간다 → 네트워크 계층에 진입 자체를 안 한다 → 관측할 게 없어서 안 뜬다.

**4. WebSocket이 fetch와 다른 지점(핵심).**

- WebSocket 연결은 처음 핸드셰이크만 HTTP다(`Upgrade` 요청 → `101 Switching Protocols`). 이 단계는 Service Worker의 `fetch` 이벤트에 잡히기도 한다.
- 그러나 `101` 이후의 실제 데이터(프레임)는 브라우저의 WebSocket 엔진이 직접 처리하고, TCP 스트림으로 흐르며 Service Worker를 우회한다.
- 그래서 프레임을 가로챌 지점이 없다. fetch는 "가로채는 자리(SW fetch 이벤트)"와 "응답 합성 수단(respondWith)"이 둘 다 있는데, WebSocket 프레임 단계엔 둘 다 없다.
- 정리: **fetch는 "응답(자기완결 객체)을 합성"하면 되고, WebSocket은 "살아있는 양방향 연결(파이프)을 합성"해야 하는데 후자를 만들 수단이 브라우저 표준에 없다.**

**5. `101`은 "연결 완료"가 아니라 "출발 신호"다.**

- 핸드셰이크 문자열(`101`)만 흉내 낸다고 연결이 성사되지 않는다.
- `101`은 "이 뒤부터 진짜 프레임이 흐른다"는 약속이고, 그 뒤에 진짜 파이프가 없으면 연결은 그 상태에서 멈춘다.
- 파이프가 성립하려면 반대편에 실제로 받아주는(listen하는) 상대가 필요하다 → 결국 서버가 필요하다.

**6. 왜 "서버 없이 + 네트워크 탭"이 동시에 안 되는가 (사슬).**

- 서버를 없앤다 → 상태를 들 곳이 브라우저 메모리밖에 없다 → 메모리 데이터는 네트워크 계층을 안 거친다 → 네트워크 탭에 안 뜬다.
- 반대로 네트워크 탭에 뜨게 하려면 → 데이터가 계층을 통과해야 한다 → 통과하려면 나갈 상대(서버)가 있어야 한다.
- 이 두 조건이 한 사슬로 묶여 있어서, 하나를 잡으면 다른 하나가 풀린다.

**7. Chrome DevTools Protocol(CDP)로도 안 되는 이유.**

- CDP는 command(클라이언트 → 브라우저)와 event(브라우저 → 클라이언트)로 나뉜다.
- `Network.webSocketFrameReceived` 같은 건 전부 **event**다. 클라이언트는 event를 구독(수신)만 할 수 있고, 발생시킬 수 없다.
- Network 도메인의 command 목록(enable/disable, 쿠키·캐시 조작, 헤더 조작, throttling, `replayXHR`, 가로채기 응답 등) 어디에도 "가짜 WebSocket 프레임을 발생시키는 command"가 없다.
- 대조: `replayXHR`(HTTP 재요청)은 있는데 WebSocket 대응물은 없다. Fetch 도메인의 `fulfillRequest`도 "이미 발생해서 멈춰 있는 진짜 요청"에 응답하는 것이라, 없는 요청을 지어내는 게 아니다.
- 즉 CDP가 "발생"시키는 건 전부 "진짜를 일으키는 것"이지 "가짜를 그리는 것"이 아니다. 관측용 프로토콜이라 애초에 가짜 발생 기능이 없다.

**8. 관측을 못 하니 경쟁 도구들이 택한 차선책.**

- MSW(WebSocket): Service Worker로 소켓을 못 잡는 걸 인정하고, 전역 `WebSocket`을 patch한 뒤 **콘솔 로그**로 보여준다(`[MSW] ⬆ ... ` 형식). 저자 본인이 "not ideal"이라고 표현.
- mock-socket: 관측 기능 없음(개발자가 알아서).
- Playwright `routeWebSocket`: 가로채는 순간 네트워크 탭에서 사라진다는 걸 그대로 받아들임.
- 결론: "서버 없이 + 기본 네트워크 탭"은 특정 도구의 게으름이 아니라 업계 전체가 넘지 못한 구조적 한계다.

**9. 곁가지로 확인한 것들.**

- 브라우저 JS는 보안 샌드박스 때문에 TCP listen(포트 열고 연결 받기)이 안 된다 → 브라우저 안에 WebSocket 서버를 못 만든다.
- Direct Sockets API(`TCPServerSocket`)로 브라우저가 서버가 될 수는 있으나 Isolated Web App(IWA) 환경 한정이라 "아무 웹앱"에선 못 쓴다.
- WebRTC는 서버 없이 peer 직접 연결이 되지만, 관측이 네트워크 탭이 아니라 `chrome://webrtc-internals`라는 별도 창구이고 메시지 내용도 안 보인다.
- BroadcastChannel/MessageChannel은 서버 없이 브라우저 안 통신이 되지만(심지어 "발신자 제외 broadcast"는 socket.io broadcast와 시맨틱이 같다) 메모리/IPC 레벨이라 네트워크 탭에 안 뜬다.
- F12에 새 탭을 추가하려면 크롬 확장(`chrome.devtools.panels.create`)이 필요하다. React DevTools의 Components 탭이 그 방식이다.

## 틀렸거나 부족했던 부분

- 처음에 "네트워크 탭 = 서버 왕복 기록"이라고 이해한 게 틀렸다. 정확히는 "네트워크 계층 통과 기록"이다. 이 오해 때문에 MSW가 왜 뜨는지 한참 설명을 못 했다.
- "핸드셰이크만 흉내 내면 연결된 것"이라고 착각했다. `101`이 출발 신호일 뿐이라는 걸 몰랐다.
- CDP를 처음엔 "붙이면 뭐든 조작 가능한 만능 통로"로 오해했다. command와 event의 방향이 고정돼 있다는 걸 몰랐다.
- 아직 부족한 부분: TCP/WebSocket 프레임 포맷(opcode, 마스킹)의 세부, Service Worker 생명주기, QUIC/HTTP3 기반 WebTransport의 동작은 개념만 훑었고 깊이는 얕다.

## 면접용 1분 답변

Q. "mock 라이브러리인데 개발자도구 네트워크 탭에 요청이 뜨게 하려면 어떻게 해야 하나요?"

A. 네트워크 탭은 서버 도달 여부가 아니라 브라우저 네트워크 계층 통과 여부를 보여줍니다. 그래서 MSW 같은 HTTP mock은 앱이 실제 `fetch`를 호출하게 두고, 네트워크 계층에 진입한 요청을 Service Worker가 가로채 `respondWith`로 가짜 응답을 돌려줍니다. 서버엔 안 갔지만 계층 진입은 실제로 일어나 네트워크 탭에 뜨죠. 반면 WebSocket은 핸드셰이크(HTTP)까지만 Service Worker가 볼 수 있고, 그 이후 프레임은 브라우저 엔진이 TCP로 직접 처리해 Service Worker를 우회합니다. fetch는 자기완결적인 응답 객체를 합성하면 되지만 WebSocket은 살아있는 양방향 연결을 합성해야 하는데, 그 수단이 브라우저 표준에 없습니다. 그래서 서버 없는 WebSocket mock은 콘솔 로깅 정도가 현실적인 관측 수단이고, 실제로 MSW도 WebSocket은 그렇게 처리합니다.

## 한 줄 요약

- 네트워크 탭에 뜨는 조건은 "서버 도달"이 아니라 "네트워크 계층 통과"이고, HTTP는 Service Worker로 계층 진입 후 가로채기가 되지만 WebSocket 프레임은 그 가로채는 지점 자체가 없어서 "서버 없이 네트워크 탭에 표시"가 현재 브라우저 표준에선 성립하지 않는다.
