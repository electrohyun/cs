## 답을 본 뒤 알게 된 것

### 핸드셰이크 (일반 용어)

- 본 통신 전에 양쪽 엔드포인트가 통신 파라미터를 합의하는 절차를 가리키는 일반 용어. 특정 프로토콜 이름이 아님
- 계층마다 하나씩 존재 — TCP 3-way handshake, TLS handshake, WebSocket opening handshake
- `wss://`면 TCP → TLS → WebSocket 순서로 3중으로 쌓여 있는 상태

### WebSocket opening handshake (RFC 6455 §4)

- 정식 명칭이 opening handshake. 대칭으로 closing handshake도 있음 (Close 프레임 `0x8` 교환)
- 클라이언트가 보내는 첫 요청은 문법상 완전한 HTTP/1.1 GET 요청
- 서버가 `101 Switching Protocols`로 답하면, **101 응답의 마지막 바이트 직후부터** 같은 TCP 연결 위 바이트 해석 규칙이 HTTP 메시지 → WebSocket 프레임으로 바뀜
- 연결을 새로 맺는 게 아님. TCP 연결 그대로 유지, TLS도 재협상 안 함

### WebSocket과 HTTP의 관계

- 둘 다 application layer(TCP 위)의 **독립 프로토콜**. 상하 관계 아니고 WebSocket이 HTTP의 하위 기능도 아님
- RFC도 별개 — HTTP/1.1은 RFC 9110 계열, WebSocket은 RFC 6455. 프레임 포맷·상태 모델·수명주기 전부 자체 정의
- HTTP가 관여하는 건 opening handshake **한 번뿐**. 101 이후엔 메서드·상태코드·헤더 개념 자체가 없어지고 요청/응답 구조도 없음
- HTTP를 진입로로 쓰는 이유: 80/443 포트, 기존 프록시·방화벽, TLS 인프라를 그대로 통과하기 위해
- 분리 증거: HTTP/2엔 Upgrade가 없어서 RFC 8441의 Extended CONNECT(`:protocol = websocket`)로 진입. HTTP/3은 RFC 9220. **진입 방식은 HTTP 버전 따라 바뀌지만 101 이후의 WebSocket 본체는 그대로**

### Upgrade 헤더

- 현재 TCP 연결에서 쓸 애플리케이션 프로토콜을 교체하자는 **제안**. 전환을 확정하는 건 Upgrade 헤더가 아니라 `101` 상태 코드
- 서버가 거절하면 일반 HTTP 응답으로 답하고 연결은 계속 HTTP
- `Connection: Upgrade`가 세트인 이유: `Upgrade`는 **hop-by-hop 헤더**라 HTTP/1.1 규정상 `Connection`에 나열해 명시해야 함. 하나라도 빠지면 규격 위반이고 프록시가 잘못 중계
- 두 값 모두 대소문자 구분 없음. 브라우저는 소문자 `websocket`으로 보냄
- 이름의 유래: RFC 2616(1999)에서 "HTTP 버전 상승"(HTTP/1.1 → 2, 평문 → TLS)을 주 시나리오로 보고 지은 이름. 실제 의미는 upgrade가 아니라 **switch**이고, 정식 명칭도 `101 Switching Protocols`

#### Upgrade의 실제 사용 현황

| 전환            | 실제 방식                           | 101 사용                  |
| --------------- | ----------------------------------- | ------------------------- |
| HTTP/1.0 → 1.1  | 요청 라인 버전 표기                 | X                         |
| 1.1 → 2 (TLS)   | ALPN                                | X                         |
| 1.1 → 2 (평문)  | `Upgrade: h2c`                      | O — **RFC 9113에서 폐기** |
| 2 → 3           | Alt-Svc / DNS HTTPS 레코드, 새 연결 | X                         |
| 1.1 → WebSocket | `Upgrade: websocket`                | **O — 유일한 실사용**     |

- h2c 폐기 이유: 브라우저 중 구현한 게 없었고, TLS면 ALPN이 추가 RTT 0으로 끝나는데 Upgrade는 왕복이 한 번 더 듦
- 역설: Upgrade가 **원래 노린 용도에서는 전부 도태되고, 부수적으로 열어둔 임의 프로토콜 전환에서만 살아남음**

### 핸드셰이크가 하는 일 3가지

1. **프로토콜 전환 합의** — 서버가 이 연결을 WebSocket으로 쓸 의사가 있는지 확인
2. **파라미터 협상** — 버전(`Sec-WebSocket-Version: 13`), 서브프로토콜, 확장(permessage-deflate)
3. **상대가 RFC 6455를 실제 구현했는지 검증** — `Sec-WebSocket-Key` / `Sec-WebSocket-Accept` 쌍

### Sec-WebSocket-Key / Accept

- **Key** = 클라이언트가 매 연결마다 생성하는 16바이트 난수를 base64 인코딩한 값. 항상 24자
  - 브라우저가 자동 생성. `Sec-` 접두사는 forbidden header라 JS로 설정 불가
- **Accept** = 서버의 계산 결과
  1. 받은 Key 뒤에 GUID 상수 `258EAFA5-E914-47DA-95CA-C5AB0DC85B11` 이어붙임 (RFC 6455에 하드코딩, 매직 스트링)
  2. SHA-1 해시
  3. 결과 20바이트를 base64 인코딩
- 클라이언트는 같은 계산을 해서 비교하고, 다르면 연결을 끊음
- **목적은 인증이 아님**. Key가 평문이고 계산 규칙·GUID 모두 공개라 누구나 계산 가능. 비밀이 하나도 없음
  - 실제 목적: 공격자가 WebSocket을 모르는 일반 HTTP 서버나 캐시에 요청을 던져 101처럼 보이는 응답을 얻어내는 **cross-protocol attack** 방지. 저 계산을 정확히 하는 건 RFC 6455 구현체뿐
  - 부가 효과: Key가 매번 난수라 프록시가 핸드셰이크 응답을 캐싱해 재사용하는 게 불가능
- SHA-1이어도 문제없음. 암호학적 안전성이 아니라 "규칙대로 계산했음"을 보이는 결정적 변환 용도라 충돌 저항성이 불필요

### 클라이언트 요청 헤더

```
GET /socket.io/?EIO=4&transport=websocket HTTP/1.1
Host: pollycasso.example.com
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ==
Sec-WebSocket-Version: 13
Origin: https://pollycasso.example.com
Sec-WebSocket-Protocol: chat, superchat
Sec-WebSocket-Extensions: permessage-deflate; client_max_window_bits
```

- **메서드는 GET 고정, HTTP/1.1 이상 고정.** POST 불가, HTTP/1.0 불가
- 경로는 자유. 서버가 이 경로로 라우팅 (Socket.IO의 `/socket.io/`가 그 예)

**필수**

| 헤더                    | 값                   | 역할            |
| ----------------------- | -------------------- | --------------- |
| `Host`                  | 호스트               | HTTP/1.1 필수   |
| `Upgrade`               | `websocket`          | 전환 대상 지정  |
| `Connection`            | `Upgrade`            | hop-by-hop 명시 |
| `Sec-WebSocket-Key`     | 16바이트 난수 base64 | 구현 검증용     |
| `Sec-WebSocket-Version` | `13`                 | 프로토콜 버전   |

- Version이 `13`인 이유: RFC 6455가 확정한 유일한 값. 0~12는 표준화 과정에서 폐기된 드래프트 번호
- 서버가 13을 못 받으면 `426 Upgrade Required` + `Sec-WebSocket-Version: 13`으로 응답

**선택**

- `Origin` — 브라우저가 자동 부착. **WebSocket에는 CORS가 없음.** preflight도 없고 브라우저가 교차 출처를 막지도 않음
  - 따라서 출처 검증 책임이 전부 **서버**에 있음. 핸드셰이크 단계에서 직접 확인하고 허용 목록에 없으면 `403`
  - 안 하면 CSWSH(Cross-Site WebSocket Hijacking). 쿠키가 자동으로 실려 가므로 공격자 사이트가 피해자 세션으로 소켓을 열 수 있음
  - 단, 브라우저가 아닌 클라이언트는 Origin을 위조 가능 → 브라우저발 공격만 막는 방어선. 진짜 인증은 별도 토큰으로
- `Sec-WebSocket-Protocol` — 서브프로토콜 협상. 프레임에 실려 가는 **메시지의 형식·의미 규약**. WebSocket 자체는 바이트 덩어리 전달까지만 정하고 해석 규칙은 안 정함
  - 예: `graphql-transport-ws`, `mqtt`, `wamp.2.json`
  - JS: `new WebSocket(url, ['graphql-transport-ws'])`
- `Sec-WebSocket-Extensions` — 프레임 레벨 확장. 실무에서 사실상 `permessage-deflate` 하나
  - 서브프로토콜과 층이 다름: 서브프로토콜 = 메시지 **내용의 의미**, 확장 = 프레임을 전송선에서 **어떻게 가공할지**. 그래서 확장은 복수 적용 가능

### 서버 응답

```
HTTP/1.1 101 Switching Protocols
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Accept: s3pPLMBiTxaQ9kYGzzhZRbK+xOo=
Sec-WebSocket-Protocol: chat
Sec-WebSocket-Extensions: permessage-deflate
```

- **101만 성공.** 2xx도 성공이 아님. 200이 오면 클라이언트는 실패로 처리하고 연결을 끊음
- **본문 없음.** `Content-Length`도 `Transfer-Encoding`도 없고, 헤더 끝 `\r\n\r\n` 다음 바이트부터가 WebSocket 프레임 — 여기가 전환의 정확한 경계선
- 필수 헤더 3개: `Upgrade`, `Connection`, `Sec-WebSocket-Accept`. 클라이언트가 셋 다 검증
- `Sec-WebSocket-Protocol`은 요청에 나열된 것 중 **정확히 하나만** 반환 가능. 요청에 없던 값을 주면 실패 처리. 생략하면 "서브프로토콜 없이 진행"
- `Sec-WebSocket-Extensions`도 요청에 제시된 것의 부분집합만. 이쪽은 복수 가능

**실패 케이스**

| 상태 코드     | 원인                                                                  |
| ------------- | --------------------------------------------------------------------- |
| `400`         | 필수 헤더 누락, Key 형식 오류. 가장 흔함                              |
| `401` / `403` | 인증 실패, `Origin` 검증 거절                                         |
| `404`         | 경로가 WebSocket 엔드포인트가 아님. 라우팅 설정 누락                  |
| `426`         | 버전 불일치. `Sec-WebSocket-Version: 13`을 같이 보내 지원 버전 통보   |
| `3xx`         | 리다이렉트. 클라이언트가 따라갈 의무 없고 브라우저는 대부분 실패 처리 |
| `200`         | 프록시·LB가 Upgrade를 이해 못 하고 일반 HTTP로 처리한 경우            |

- 마지막 케이스가 실전 최다 함정. nginx 기본 설정은 `Upgrade`를 전달하지 않음(hop-by-hop이라 프록시가 소비) → `proxy_set_header Upgrade $http_upgrade;` + `proxy_set_header Connection "upgrade";` 명시 필요. **`Connection: Upgrade`가 hop-by-hop이라는 사실이 여기서 실제 비용으로 나타남**

**클라이언트 측 관찰 제약**

- 브라우저 JS에서 핸드셰이크 실패 사유를 구분할 방법이 사실상 없음
- `error` 이벤트에 상태 코드·실패 사유가 안 담기고, `close` 이벤트의 `code`는 `1006`(abnormal closure)으로 뭉뚱그려짐
- 스펙이 의도한 제약 — 실패 정보를 JS에 노출하면 포트 스캐닝에 악용 가능
- 실패 원인은 Network 탭이나 서버 로그로만 확인 가능

### 101 이후

- 요청-응답 짝 개념 없음. 양쪽 다 아무 때나 메시지 전송 가능
- **클라이언트 → 서버 프레임은 반드시 마스킹**(4바이트 마스킹 키로 XOR). 서버 → 클라이언트는 마스킹 안 함. 이것도 보안이 아니라 캐시 오염 공격 방지용
- 종료는 closing handshake — 한쪽이 Close 프레임(opcode `0x8`) 전송 → 상대가 Close 프레임 응답 → TCP 종료

## 틀렸거나 부족했던 부분

- **"WebSocket이 HTTP의 기능"으로 이해하고 있었음** → 아님. 같은 계층의 독립 프로토콜이고, HTTP는 진입로로만 쓰임. HTTP/2·3에서 진입 방식이 바뀌어도 WebSocket 본체는 그대로라는 게 증거
- **Upgrade의 "up"을 우열로 읽었음** → 1999년 "HTTP 버전 상승"을 주 시나리오로 본 작명일 뿐. 실제 의미는 switch. HTTP가 열등한 게 아니라 애초에 실시간 양방향용으로 설계된 적이 없는 것
- **`Sec-WebSocket-Key`를 보안·인증 장치로 오해하기 쉬움** → 평문이고 계산 규칙 공개라 누구나 계산 가능. 구현 검증 + 캐시 무력화 용도
- **전환 확정 주체를 `Upgrade` 헤더로 오해하기 쉬움** → 확정하는 건 `101` 상태 코드
- **`Connection: Upgrade`를 관용구로 외우고 있었음** → hop-by-hop 헤더 명시 규정이 이유이고, 이게 nginx 프록시 설정 문제로 직결됨

### 아직 남은 갭

- 프레임 구조 (opcode, 마스킹, 페이로드 길이 인코딩)
- Socket.IO / Engine.IO가 이 위에 얹은 것 — 폴링 폴백, 재연결, ack
- nginx WebSocket 프록시 설정 실물
- 개발자도구 Network 탭에서 실제 101 응답 확인 (폴리카소)

## 면접용 1분 답변

> WebSocket 핸드셰이크는 HTTP/1.1의 Upgrade 메커니즘을 진입로로 써서 프로토콜을 전환하는 절차입니다.
>
> 클라이언트가 `Upgrade: websocket`, `Connection: Upgrade`, `Sec-WebSocket-Key`, `Sec-WebSocket-Version: 13`을 담은 GET 요청을 보내고, 서버가 `101 Switching Protocols`로 응답하면 그 시점부터 같은 TCP 연결 위의 바이트가 HTTP 메시지가 아니라 WebSocket 프레임으로 해석됩니다. 연결을 새로 맺는 게 아니라 해석 규칙만 바뀌는 겁니다.
>
> HTTP를 진입로로 쓰는 이유는 80·443 포트와 기존 프록시·TLS 인프라를 그대로 통과하기 위해서고, WebSocket 자체는 HTTP의 하위 기능이 아니라 같은 계층의 독립 프로토콜입니다. HTTP/2에서는 101 대신 Extended CONNECT로 진입 방식이 바뀌지만 그 이후의 프로토콜 본체는 동일하다는 게 이 분리를 보여줍니다.
>
> `Sec-WebSocket-Key`와 `Accept`는 인증이 아니라, 고정 GUID를 붙여 SHA-1·base64한 값을 서버가 정확히 돌려주는지 확인해서 상대가 실제 WebSocket 구현체인지 검증하는 장치입니다. cross-protocol attack과 프록시 캐싱을 막습니다.
>
> 실무에서는 `Upgrade`가 hop-by-hop 헤더라 nginx 같은 리버스 프록시가 기본 설정에서 이 헤더를 전달하지 않아 101이 안 오고 200이 오는 문제를 자주 만납니다.

## 한 줄 요약

- WebSocket 핸드셰이크는 HTTP/1.1의 Upgrade 메커니즘으로 진입해 `101`로 전환을 확정하는 절차이고, 그 안의 Key/Accept·Origin·서브프로토콜·확장은 각각 구현 검증·출처 검증·메시지 의미 협상·프레임 가공 협상이라는 서로 다른 층의 일을 한다.
