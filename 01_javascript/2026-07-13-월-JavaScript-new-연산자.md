# JavaScript의 new 연산자

## 잘 알고 있는지

## 내가 아는 만큼 설명

- 오늘 알고리즘 루틴에서 4일차 new Array를 하면서 떠오른 CS 질문이다.
- C++에서는, new를 사용하면 새로운 객체를 만들고 그 객채의 주소를 반환한다.
- JS도 동일할까?
- 예상: 객체를 만드는 건 같을 듯
  → 오늘 확인할 것: JS new의 정확한 단계, 그리고 C++엔 없는 단계가 있는지

## 답을 본 뒤 알게 된 것

```js
const arr = [1, 2, 3];
```

- arr.push와 arr.map(...) 이거 왜 될까? -> 클래스의 메서드를 쓴 느낌. 정확히는 뭔지 모름.
- 사실, [1,2,3]이 Array의 인스턴스임. new Array가 도는 것

```
arr.push 하면 JS가 이 순서로 찾아:

arr 안에 push 있나? → 없음 (숫자 셋뿐)
없으면 arr에 몰래 걸려있는 링크를 타고 Array.prototype으로 감
거기 push 있음 → 그거 씀
```

- 정리하면, 객체는 자기한테 없는 것을 프로토타입 링크 타고 찾아옴
- 여기서 프로토타입이란, "없으면 대신 찾아볼 객체" 정도로만 이해 (추후 또 CS 공부주제로!!)
- arr에 몰래 링크를 걸어놓은 사람 = new (new Array가 속으로 돌면서 링크 걸어줌)
- 즉 "프로토타입 연결"은 new가 하는 일 / 근데 이것만 하는가?

- 제일 먼저 new가 하는 일은 빈 객체를 새로 하나 만드는 일이다. 즉, 메모리 공간을 할당한다. 이게 우선이고, 다음이 링크. 이 객체를 프로토타입에 링크.

=====

```js
function Dog(name) {
  this.name = name; // ← 얘를 아직 안 함
}
const d = new Dog("두부");
```

- 질문. 원래 문장 구조는 `const 인스턴스이름 = new 클래스이름(인자)` 아닌가요? -> JS에서 class는 나중에 추가된 겉 포장이다.

```js
// 원조 방식 (생성자 함수)
function Dog(name) {
  this.name = name;
}

// 겉포장 (class 문법, 2015년에 추가됨)
class Dog {
  constructor(name) {
    this.name = name;
  }
}
```

- 둘 다 new Dog("두부")로 쓰고, 둘 다 결과는 같다.
- `class`는 `function`을 보기 좋게 감싼 문법적 설탕(symentic sugar)이다.
- 속에서는 여전히 함수가 돌고, 프로토타입에 링크를 걸며, 지금 배우는 4단계가 똑같이 돈다. (빈 객체 → 링크 → this 바인딩 → 반환)

- JS에서는 `new`가 클래스 전용이 아니다. `new`는 함수에 붙이는 연산자이다.

- `const 인스턴스 = new 생성자함수(인자)`
  - class든 function이든 속은 "생성자 함수". new는 함수에 붙는 연산자.

=====

- Q. d가 {name: "두부"}가 되려면 this가 뭐와 같아야 할까?

- A. 새로 만든 객체 - 즉 자기 자신. -> 90%만 정답

- 빈 객체 {} 생성,
  걔를 Dog.prototype에 링크,
  this를 그 {}에 묶고 Dog 본문 실행 → this.name = "두부" → {}가 {name:"두부"}로 채워짐 ← 방금 네가 맞힌 거

=====

- Q. `Dog` 함수 안에 `return`이 없는데, new Dog("두부")는 어떻게 돌려줬는가?

- 생각: `new`가 해주지 않았을까?

- A. 본문에서 return 안 하거나, 원시값(숫자·문자열 등)을 return하면 → 3번 객체가 반환 (위 경우)
  본문에서 객체를 명시적으로 return하면 → 그 객체가 대신 반환됨 (3번 무시)

=====

- Q. 마지막 확인 질문. JS에만 있고 C++에는 없는, 그 단계가 있어야만 하는 이유

- A. 2번 링크 과정. JS 객체는 자기 자신한테 없는거를 찾기 위해서, 즉, 프로토타입이 필요해.

```
정확해. 그게 핵심이야.
C++은 클래스에 메서드가 박혀서 컴파일돼. 객체가 어디서 메서드를 "찾을" 필요가 없어 — 이미 자기 타입에 소속돼 있으니까. 그래서 링크 걸 단계가 필요 없지.
JS는 반대야. 객체한테 메서드가 안 박혀있어. arr은 숫자 셋만 든 빈 껍데기고, push는 딴 데(Array.prototype) 있어. 그러니 "없으면 어디서 찾을지"를 알려주는 링크가 반드시 있어야 객체가 쓸모있어져. 그 링크 거는 게 2번이고, 그래서 2번은 JS에만 있어. 네가 정확히 짚었어.
```

=====

- 총정리, 코드로 다시보기

```js
const a = [1, 2, 3];
const b = [9, 8, 7];

a.push(4);
b.push(0);
```

```
Array            ← 생성자 함수 (new 붙이는 그거)
Array.prototype  ← 메서드 창고 객체. push가 여기 딱 1개 들어있음.
a, b             ← new Array로 찍은 배열들. 각자 창고에 링크만 걸림.
```

- a.push(4)를 하면 벌어지는 일?

```
a 안에 push 있나? → 없음 (1,2,3,4 숫자뿐)
   ↓ a에 걸린 링크 타고 감
Array.prototype 도착 → push 있음 → 실행
```

```
- push는 Array.prototype에 1개. 모든 배열이 링크로 공유해서 빌려 씀.
- 링크는 "메서드가 사는 창고"(Array.prototype)로 걸림.
  Array(함수)로 걸리는 게 아님 — 함수엔 메서드가 안 살아서.
```

=====

```js
const a = [1, 2, 3];
```

```js
const a = new Array(1, 2, 3);
```

- 둘은 완전히 같다. (그러나 사소한 차이 - new Array는 인자가 하나면 그 인자만큼의 길이를 갖는 빈 배열로, 좋은 사용법이 아니다. quirk)

1. a라는 빈 객체를 만든다. 메모리를 확보한다.
2. Array로 만들었네? Array.prototype과 연결. 이해가 안되어서 질문.

- "new Array니까 new가 "아 생성자가 Array구나 → 그럼 찾아갈 창고는 Array.prototype이지" 하고 그 링크를 걸어. 생성자가 Array면 링크는 자동으로 Array.prototype으로 감. Dog였으면 Dog.prototype으로 갔을 거고. 이 순간부터 a.push가 가능해져 — 링크가 생겼으니까. 이게 2번의 전부야."

3. 그 빈 객체에 프로퍼티가 삽입된다. 즉 생성자 함수 본문이 this에다 값을 꽂는 단계. Array의 함수 본문이. this가 1번 객체에 묶인 채 이렇게 해준다.

```js
// 3번: this = 1번의 빈 객체. Array 본문이 이걸 채움 (개념적으로)
this[0] = 1;
this[1] = 2;
this[2] = 3;
this.length = 3;

// 그리고

function Dog(name) {
  this.name = name; // ← 이 줄이 "3번: 프로퍼티 삽입"
}
new Dog("두부"); // this = 빈 객체 → this.name="두부" → {name:"두부"}
```

4. 본문이 끝나면 new가 그 객체를 반환해 담아준다. return을 안 써도 자동 - (값이 있으면 그걸, 없으면 자동)

=====

- 이제부터 확인질문 모음.

1. 2번(링크)이랑 3번(값 삽입) 순서를 바꾸면 안 되는 이유가 있을까? 왜 링크를 먼저 걸고 값을 나중에 꽂을까? (힌트: 3번에서 this가 가리키는 게 뭐였지)

- 프로퍼티 연결은 상관없다. this[0] = 1, ... 이런 것. 그런데...
- 메서드 연결이 문제다. 생성자 본문이 this.메서드() 식으로 호출이 가능한데, 본문이 실행되는 시점(3번)에는 이미 this가 완성된 객체여야 한다.
- 깔끔히 말하면: 값이 삽입될 때- this.메서드()를 부를 수 있다. 그 메서드는 프로토타입 창고에 있는 것일수도 있다. 그러므로 사용자에게 this가 완성된 객체로 보장되어야 한다는 일종의 약속이 있는 것이다.

=====

- 진짜 제대로 된 마지막 정리(진짜)

```js
const a = new Array(1, 2, 3);
```

1. new가 빈 객체를 힙에 생성 (메모리 확보). a는 그 객체를 "참조"함. (
   - 참조 = 객체가 힙의 어디에 있는지 "가리키는 것" (변수는 객체를 담는 게 아니라 가리킴)
   - const b = a → 참조만 복사. 둘이 같은 객체 가리킴. b.push하면 a도 바뀜.
   - 원시값(숫자 등)은 값 자체가 복사 → 서로 남남.)

   - (C++과의 차이: C++은 포인터 vs JS는 참조.
   - a가 든 값 = 실제 주소 숫자 vs a가 든 값 = 참조(엔진이 감춤)
   - 그 주소를 출력/연산 가능 vs 주소 못 봄, 못 만짐
     (a+1 하면 옆 메모리로 이동) vs (a+1? 그런 거 없음)
   - delete a; 로 직접 메모리 해제 vs 해제를 안 함. GC가 알아서 치움
   - 잘못 만지면 터짐(댕글링 포인터 등) vs 만질 수가 없어서 그 사고가 원천봉쇄)

2. new가 그 객체의 [[Prototype]]을 Array.prototype으로 설정한다.
   이때부터 a.push 가능.
3. new가 this를 1번에서 생성한 객체로 설정한 상태로 생성자 본문을 실행한다.
   본문의 this.프로퍼티 = 값 은 그 객체에 프로퍼티를 추가한다.
   (프로퍼티를 추가하는 주체 = 생성자 본문.
   본문 실행 시점에 this가 완성된 객체를 참조해야 하므로 2번과 3번의 순서는 고정.)
4. new가 그 객체(참조)를 반환 → a에 담김. a는 배열로 동작.

"참조는 명세상 형태가 정의되지 않은 추상이고, 구현(V8 등)에선 대개 객체 주소를 담은 포인터라 16진수로 표현될 수 있다. 단 GC가 옮기면 바뀌고 JS에선 접근 불가."

## 면접용 1분 답변

- Q. "new 연산자로 함수를 호출하면 내부적으로 무슨 일이 일어나나요?"
- A. 4단계로 이루어집니다. 처음으로 빈 객체를 생성하고, 두번째로 빈 객체의 프로토타입 슬롯을 생성자 함수의 프로토타입 프로퍼티로 설정합니다. 이 단계 덕분에 인스턴스가 생성자의 프로토타입에 정의된 메서드를 조회해서 쓸 수 있습니다. 세번째로 this를 그 객체로 설정한 상태로 생성자 본문을 실행하고, 본문에서 this에 프로퍼티를 추가해 객체를 채워 완성합니다. 네번째로, 생성자가 객체를 명시적으로 반환하지 않으면 new가 이 객체를 반환하고, 변수는 그 객체에 대한 참조를 받습니다.

- Q. "그럼 return 해주면요?"
- A. 생성자가 객체를 명시적으로 반환하면 그 객체가 반환됩니다. 원시값을 반환하면 무시되고, 원래 생성된 객체가 반환됩니다.

## 한 줄 요약

- 오늘은 배운 게 많아서 최대한 정리해보는 걸로.
- c++과 달리 JS는 메모리로부터 추상화된 참조라는 개념을 쓴다.
- const a = new Array(1,2,3) 를 하게 되면, 4단계 과정을 거친다.

(긴 요약)

- 1. 빈 객체를 만든다. 이 객체를 위한 메모리가 힙에 할당된다.

- 2. new가 a의 [[Prototype]] 슬롯을 Array.prototype으로 설정한다.
  - [[Prototype]]: 객체마다 있는 내부 슬롯. 없는 프로퍼티를 조회할 때
    다음으로 볼 객체(의 참조)를 담는다.
  - Array.prototype: Array 함수의 prototype 프로퍼티. push/map 등이
    프로퍼티로 정의된 객체. (Array 함수 자체와는 다른 값.)
  - 설정 대상 = new 오른쪽 생성자의 .prototype.
  - 목적: 이 설정 이후 a.push 조회 시 a→Array.prototype 순으로 찾게 됨.

- 3. new가 this를 그 객체로 설정한다 = 명세상:
  - [[Construct]]가 새 객체(thisArgument = 1번의 그거)를 만든다 (OrdinaryCreateFromConstructor).
    → 이때 객체 생성 + [[Prototype]] 설정이 함께 일어남 (1·2번의 실제 위치).
  - 함수 본문 실행용 실행 컨텍스트를 만든다 (PrepareForOrdinaryCall).
    이 컨텍스트의 함수 환경 레코드에 [[ThisValue]] 슬롯이 있다.
  - OrdinaryCallBindThis → BindThisValue(thisArgument):
    envRec.[[ThisValue]] = thisArgument
    envRec.[[ThisBindingStatus]] = "initialized"
    ← "this를 그 객체로 설정"의 정확한 실체 = 이 슬롯에 객체를 저장.
  - 본문에서 this 참조 시, ResolveThisBinding이 그 슬롯을 읽어 반환.
  - this는 변수가 아니라 환경 레코드 슬롯에서 조회되는 값이다.

(- 실행 컨텍스트 = 코드가 실행될 때 필요한 상태를 담는 명세상 레코드. ( - DB record: 테이블에 저장되는 데이터 한 행. 사용자 데이터. 디스크/메모리에 실재. - 명세 record: 명세가 엔진 내부 구조를 "필드들의 묶음"으로 서술하기 위한 기술 용어. 실제 구현이 뭘로 만들지는 엔진 자유. 명세상의 서술 장치. - 이름만 같고 별개 개념.))
(전역/함수/모듈/eval 실행마다 하나씩 생성)

- 담는 것:
  - LexicalEnvironment: 식별자(let/const/함수 등)를 조회하는 환경 레코드 참조
  - VariableEnvironment: var 관련 환경 레코드 참조
  - code evaluation state: 실행 진행 지점
- 함수 실행 컨텍스트 → 함수 환경 레코드 보유 → 그 안에 [[ThisValue]] (this가 여기서 나옴)
- 실행 컨텍스트 스택(=콜 스택): 호출 시 push, 종료 시 pop. 맨 위가 현재 실행 중.)

- 4. new가 그 객체(참조)를 반환 → a에 담김. a는 배열로 동작. 생성자가 객체를 명시적으로 반환하지 않으면, new가 그 객체를 반환한다. a는 그 객체를 참조한다.

(짧은 요약)

1. 빈 객체를 생성한다. 메모리에 객체를 위한 공간이 할당된다.
2. 빈 객체의 [[Prototype]]이 new에 의해 Array.prototype에 대한 참조를 담게 된다. push, shift 등의 메서드는 이 prototype으로부터 사용할 수 있게 된다.
3. new가 this를 바인딩해 본문의 내용을 실행하고, 그 본문이 프로토타입이 연결된 빈 객체에 프로퍼티를 할당한다.
4. 이후 new가 이 객체의 참조를 반환해준다.
