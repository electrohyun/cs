## 답을 본 뒤 알게 된 것

### 1. 실행 컨텍스트 (Execution Context)

- JS 엔진은 코드를 실행할 때 실행 컨텍스트(Execution Context) 라는 내부 자료구조를 만든다. 만들어지는 시점은 두 가지
  - 스크립트가 평가될 때 → 전역 실행 컨텍스트 1개
  - 함수가 호출될 때 → 그 호출마다 1개 (함수 정의 시점이 아니다. 호출 시점이다)

- 컨텍스트 안에 들어있는 것
  - LexicalEnvironment: 매개변수, let/const, 함수 선언이 담기는 환경 레코드
  - VariableEnvironment: var가 담기는 환경 레코드
  - ThisBinding: 이 호출에서 this가 가리키는 값
  - 반환 주소: 이 함수가 끝나면 어느 코드 위치로 돌아갈지
  - Outer 참조: 상위 스코프 환경 레코드에 대한 참조

- 같은 함수를 세 번 호출하면 컨텍스트가 세 개 생긴다. 각각 자기 매개변수와 지역 변수를 따로 들고 있다. 재귀 호출에서 각 단계의 변수가 안 섞이는 이유가 이것.

#### 스택 동작

- 이 컨텍스트를 담는 자료구조가 콜 스택(Call Stack)

- LIFO 구조, 마지막에 push된 게 먼저 pop 된다.

- 함수를 호출하고, 컨텍스트를 생성한 뒤, push.
- return에 도달하거나, 함수 끝(암묵적 return undefined) 도달 시, pop, 반환 주소로 제어권 이동

- 스택의 최 상단 컨텍스트가 곧 "지금 실행 중인 코드".
- 그 아래 컨텍스트는 모두 중단 상태.
- 실행이 멈춘 게 아니라, 자기 자리에서 호출문의 결과를 기다리며 다음 줄로 못 넘어가고 있는 상태.

```js
function c() {
  return 3;
}
function b() {
  return c() + 1;
}
function a() {
  return b() * 2;
}
a();
```

```
[전역] 스크립트 평가
[전역, a] a() 호출 → a 컨텍스트 push
[전역, a, b] b() 호출. a는 b()의 결과가 없어 _ 2 를 못 함
[전역, a, b, c] c() 호출
[전역, a, b] c가 3 반환 → c pop → b의 반환 주소로 복귀
[전역, a] b가 3 + 1 = 4 반환 → b pop
[전역] a가 4 * 2 = 8 반환 → a pop
```

#### 스택 트레이스가 이 스택 트레이스다.

```js
function c() {
  throw new Error("boom");
}
function b() {
  c();
}
function a() {
  b();
}
a();
```

```
Error: boom
    at c (index.js:2:9)
    at b (index.js:5:15)
    at a (index.js:6:15)
    at index.js:7:1
```

#### 크기 제한

```js
function loop() {
  return loop();
}
loop(); // RangeError: Maximum call stack size exceeded
```

- 스택은 크기가 정해져 있다. pop 없이 push만 계속되면 넘친다.

#### 그리고, 오늘의 전제

- 콜 스택에 컨텍스트가 하나라도 남아 있는 동안, 다른 코드는 실행될 수 없다.

- JS 실행 스레드는 하나고, 그 스레드가 하는 일이 곧 이 스택을 처리하는 것.
- 스택 최상단 함수가 100ms 걸리는 계산을 하고 있으면 그 100ms 동안 스레드는 거기 묶여 있다.
- 타이머가 끝나든 클릭이 발생하든 fetch 응답이 오든, 그 콜백은 지금 실행될 방법이 없다. 실행할 스레드가 없으므로...

- 그러면 그 콜백들은 그동안 어디에 있고, 언제 스택에 올라오는가?
- 이게 다음 내용들이다. 그리고 프로미스 1일차에 적어둔 "코드 끝 → 엔진이 대기줄 꺼내 실행"의 그 대기줄이 여기서 나온다.

### 2. 런타임 구성

#### 엔진이 가진 것

- JS 엔진(V8, SpiderMonkey 등)이 제공하는 것은 두 개다.
  - 콜 스택: 실행 컨텍스트를 쌓는 곳
  - 힙: 객체가 할당되는 메모리 영역

- 그리고 ECMAScript 명세에 정의된 내장 객체들: Object, Array, Function, Promise, JSON, Math, Map, Set 등.

- 그런데... 여기에 없는 것들이 있다. 많이 써봤는데도. 이것들은 ECMAScript 명세에 없다. 엔진은 이들을 모른다.

```
setTimeout, setInterval
fetch, XMLHttpRequest
document, addEventListener
localStorage, requestAnimationFrame
```

#### 호스트 환경이 가진 것

- 이것들은 엔진이 감싸고 있는 호스트 환경이 제공한다. 브라우저면 브라우저가, Node면 Node가.

```js
// 브라우저
document.querySelector("div"); // 동작
require("fs"); // ReferenceError

// Node
require("fs"); // 동작
document.querySelector("div"); // ReferenceError
```

- setTimeout은 양쪽 다 있지만 같은 함수가 아니다. 브라우저는 HTML 명세의 WindowOrWorkerGlobalScope.setTimeout, Node는 자체 구현이고 반환값도 다르다. 브라우저는 숫자 ID, Node는 Timeout 객체를 반환한다.

- 즉 window 또는 globalThis에 붙어 있는 것 중 상당수는 JS 언어가 아니라 호스트 환경이 꽂아 넣은 것이다.

#### setTimeout 호출 시 실제로 일어나는 일

```js
setTimeout(() => console.log("B"), 1000);
```

1. setTimeout 컨텍스트가 콜 스택에 push된다.
2. 이 함수의 본체는 JS가 아니라 브라우저가 C++로 구현한 코드다. 브라우저 내부의 타이머 모듈에 (콜백 함수 참조, 만료 시각) 쌍을 등록한다.
3. 타이머 ID를 반환하고 즉시 pop된다.
4. 다음 줄이 실행된다.

- 여기서 핵심은 1000ms를 기다리는 주체가 JS 스레드가 아니라는 것이다. 콜 스택은 3번 시점에 이미 setTimeout을 비웠다. 대기는 브라우저 쪽 타이머 모듈이 별도로 처리한다.

- fetch도 구조가 같다. 요청 정보를 브라우저 네트워크 모듈에 넘기고, 프로미스 객체를 반환하고, 즉시 pop된다. 실제 소켓 통신은 브라우저의 다른 스레드에서 일어난다.

- "JS는 싱글 스레드인데 어떻게 동시에 여러 일을 하느냐"의 답이 여기다. 동시에 하는 건 브라우저고, JS 스레드는 하나다. JS 스레드는 등록만 하고 빠진다.

#### 만료된 뒤

- 1000ms가 지나면 브라우저는 그 콜백을 콜 스택에 직접 push하지 않는다.

- push해버리면 그 시점에 실행 중이던 함수의 중간에 다른 코드가 끼어들게 된다. 1번에서 확인한 전제 — 스택에 컨텍스트가 남아 있는 동안 다른 코드는 실행될 수 없다 — 가 깨진다. 지역 변수를 읽고 쓰는 도중에 다른 함수가 같은 변수를 건드릴 수 있게 되고, 스레드 하나로 돌아가는 이점이 사라진다.

- 그래서 브라우저는 콜백을 큐에 넣어 둔다. 그 큐가 뭐고 언제 꺼내는지가 3번과 4번이다.

#### 명세가 갈라져 있다

- 이벤트 루프도 엔진이 아니라 호스트 환경 소관이다.

- HTML 명세 — 이벤트 루프, 태스크 큐(task queue)
- ECMAScript 명세 — Job과 Job Queue (PromiseJobs)

- 프로미스 콜백은 ECMAScript 쪽 Job이고, setTimeout 콜백은 HTML 쪽 task다. 소속 명세가 다르다. 5번에서 두 콜백의 실행 순서가 왜 갈리는지 다룰 때 이게 근거가 된다.

#### 문제

```js
console.log("A");

setTimeout(() => console.log("B"), 0);

const end = Date.now() + 1000;
while (Date.now() < end) {} // 1초 동안 아무것도 안 하고 반복

console.log("C");
```

- 출력 순서는?
  - A C B
- setTimeout의 지연 시간이 0인데도 B가 1초 뒤에야 나온다. while 루프가 도는 동안 콜 스택에는 무엇이 있고, 그 때문에 콜백이 실행되지 못하는 이유를 스택 상태로 설명해봐.
  - 전역컨텍스트 + while함수 순으로 담겨있는 상태고, 한번에 하나만 실행되기 때문에 while이 종료되지 않았기 때문에 실행되지 못함.
  - 땡! while은 함수가 아니라 문(statement),
  - Date.now()가 호출되는 순간마다 [전역, Date.now]가 됐다가 즉시 pop되고, 다시 [전역]으로 돌아온다.
  - 나머지 논리는 맞다. 전역 컨텍스트가 pop되지 않았고 — 스크립트가 아직 안 끝났으니까 — 스레드는 그 컨텍스트 안의 while 조건 검사를 반복하는 데 묶여 있다. 타이머는 0ms 뒤 만료돼서 콜백을 큐에 넣어놨지만, 꺼내갈 스레드가 없다. 스크립트 끝까지 가서 C가 찍히고 전역 컨텍스트가 pop된 뒤에야 B가 실행된다.
  - 0이 "즉시"가 아니라 "최소 0ms 뒤, 그리고 스택이 빈 다음"이라는 뜻인 이유가 이거다.

```
[전역]                      스크립트 평가, 전역 컨텍스트 push
[전역, console.log]         console.log('A') 호출
[전역]                      pop, 'A' 출력됨
[전역, setTimeout]          setTimeout 호출
[전역]                      pop. 타이머 등록만 하고 즉시 빠짐
[전역]                      ← while 루프 도는 1초 내내 이 상태
[전역, console.log]         console.log('C') 호출
[전역]                      pop, 'C' 출력됨
[]                          스크립트 끝. 전역 컨텍스트 pop
```

- while 줄에서 스택에 새로 쌓이는 게 없다. Date.now()가 호출될 때만 잠깐 [전역, Date.now]가 됐다가 바로 [전역]으로 돌아온다.

- 그런데 왜 B가 못나오냐면, 이유는 전역 컨텍스트가 아직 pop되지 않았기 때문이다. 위 표에서 마지막 []가 되기 전까지, 스택은 계속 비어있지 않다.

- 타이머는 0ms 뒤에 만료됐다.
- 브라우저가 콜백을 큐에 넣었다.
- 근데 그 콜백을 스택에 올리려면 스택이 비어야 한다.
- 스택엔 전역 컨텍스트가 있다. 스크립트가 안 끝났으니까.
- while이 1초 동안 스크립트를 못 끝내게 붙잡고 있다.

- 그래서 C까지 찍히고 스크립트가 완전히 끝나서 스택이 []가 된 뒤에야 B가 올라온다.

### 3. 태스크 큐

#### 태스크란

### 태스크란

- 호스트 환경이 "이 함수를 나중에 실행해달라"고 등록해둔 작업 단위를 **태스크(task)** 라고 한다. HTML 명세 용어이고, 흔히 매크로태스크라고도 부르지만 명세 용어는 task다.

- 태스크 소스: 태스크가 큐에 들어갈 때 붙는, 발생 출처의 분류 이름.
- 브라우저가 소스별로 큐를 따로 두고 우선순위를 다르게 주기 위해 존재한다.

| 태스크 소스   | 언제 큐에 들어가는가                   |
| ------------- | -------------------------------------- |
| 타이머        | `setTimeout` / `setInterval` 만료 시점 |
| DOM 이벤트    | 클릭, 키입력 등이 발생한 시점          |
| 네트워크      | 응답이 도착한 시점                     |
| 스크립트 평가 | `<script>` 태그를 만난 시점            |

- 그리하여, 지금까지 본 "동기 코드"도 하나의 태스크.
- 스크립트 전체가 태스크 하나로 실행되고, 그게 끝나야 큐에 있는 다음 태스크가 올라온다.
- 2번에서 while 루프가 B를 막았던 게 이거다. 스크립트라는 태스크가 안 끝난 것이다.

#### 큐에 들어가는 것

```js
function log() {
  console.log("B");
}
setTimeout(log, 1000);
```

- 큐에 들어가는 건 함수 실행 결과가 아니라 함수 참조.

```
{ callback: log(함수 참조), expiry: 현재시각 + 1000 }
```

- setTimeout 호출 시점에 브라우저 타이머 모듈에 저장되는 것

```
태스크 큐: [log]
```

- 1000ms가 지나면 타이머 모듈이 이 항목을 지우고, 태스크 큐에 log를 넣는다.
- 이 시점에도 log는 아직 실행되지 않았다. 큐에 들어가 있을 뿐이다. 실행은 스택이 빈 뒤에 일어난다.

#### FIFO

```js
setTimeout(() => console.log("A"), 0);
setTimeout(() => console.log("B"), 0);
setTimeout(() => console.log("C"), 0);
console.log("sync");
```

```
큐: [A]
큐: [A, B]
큐: [A, B, C]
```

- 셋 다 0ms라 거의 동시에 만료되지만, 만료 처리는 등록 순서대로 되므로 큐에도 그 순서로 들어간다.
- 스크립트 태스크가 끝난 뒤 앞에서부터 하나씩 꺼내진다. 출력은 sync A B C.

#### 등록 순서 ≠ 큐 진입 순서

```js
setTimeout(() => console.log("A"), 100);
setTimeout(() => console.log("B"), 0);
```

- 큐에 들어가는 시점은 등록 시점이 아니라 만료 시점이다.

```
{ callback: A, expiry: t+100 }
{ callback: B, expiry: t+0 }
```

- A가 먼저 등록됐지만 타이머 모듈에 저장된 만료 시각이 다르다. t+0에 B가 큐에 들어가고, t+100에 A가 들어간다. 출력은 B A.

#### 지연 시간은 최솟값이다

- setTimeout(f, 100)이 보장하는 것은 "100ms 뒤 실행"이 아니라 "100ms 전에는 큐에 들어가지 않음" 이다. 큐에 들어간 뒤 얼마나 기다릴지는 앞에 뭐가 쌓여 있는지, 스택이 언제 비는지에 달려 있다.

```js
setTimeout(() => console.log("B"), 100);

const end = Date.now() + 3000;
while (Date.now() < end) {}
```

- B는 100ms 시점에 큐에 들어가지만 실제 실행은 3000ms 이후다.

#### 0은 0이 아니다

- HTML 명세에는 중첩 레벨 규정이 있다. setTimeout 안에서 다시 setTimeout을 호출하는 깊이가 5를 넘으면, 지연 시간이 4ms 미만이어도 4ms로 고정된다.

```js
let n = 0;
function tick() {
  n++;
  if (n < 10) setTimeout(tick, 0);
}
setTimeout(tick, 0);
```

- 이 코드에서 처음 5회는 거의 즉시 돌지만, 6회차부터는 회당 최소 4ms가 걸린다. setTimeout(f, 0)을 반복 호출하는 방식으로 애니메이션을 만들면 안 되는 이유다. 이 용도에는 requestAnimationFrame을 쓴다.

#### 큐는 하나가 아니다

- 명세상 태스크 큐는 여러 개이고, 태스크 소스별로 분리되어 있다. 타이머 큐, 사용자 상호작용 큐, DOM 조작 큐 등.

- 이벤트 루프는 매 반복마다 그중 어느 큐에서 꺼낼지 고른다. 어떤 기준으로 고르는지는 명세가 정하지 않고 브라우저 구현에 맡긴다. 크롬은 사용자 입력 큐에 우선순위를 준다. 클릭 반응이 타이머보다 늦으면 체감 성능이 나빠지기 때문이다.

- 명세가 보장하는 것은 같은 큐 안에서의 FIFO 순서뿐이다. 서로 다른 소스에서 나온 태스크의 상대 순서는 보장되지 않는다. setTimeout끼리는 순서가 보장되고, setTimeout과 클릭 이벤트 사이는 보장되지 않는다.

#### 관측 불가능

- 태스크 큐는 JS에서 접근할 수 없다. 브라우저 내부 자료구조이고 전역에 노출된 객체가 없다. 큐에 몇 개가 대기 중인지 확인하는 방법도 없다.

### 4. 이벤트 루프 1회 반복

#### 이벤트 루프

- 지금까지 "콜 스택이 비면 태스크 큐에서 꺼낸다"고 했는데, 그걸 실제로 하는 주체를 아직 안 정했다. 그게 **이벤트 루프(Event Loop)** 다.

- JS 코드가 아니다. 브라우저가 구현해서 계속 돌리고 있는 반복문이고, 페이지가 열려 있는 동안 멈추지 않는다. 하는 일은 한 가지다. 큐를 확인하고, 조건이 맞으면 하나 꺼내서 콜 스택에 올린다. 동작을 의사코드로 쓰면 이렇다.

```
while (브라우저가 살아있는 동안) {
  if (콜 스택이 비어있지 않음) continue;   // 아무것도 안 함

  task = 태스크 큐에서 하나 꺼냄;
  if (task 없음) continue;

  task를 콜 스택에 push;
  task가 끝나서 콜 스택이 다시 빌 때까지 대기;

  // ← 5번에서 여기에 한 단계가 더 들어간다
}
```

- 한 바퀴에 꺼내는 태스크는 **정확히 하나**다. 큐에 10개가 쌓여 있어도 한 바퀴에 하나만 처리하고, 다음 바퀴에서 다음 걸 꺼낸다. 이 "한 바퀴"를 **틱(tick)** 이라고 부른다.

#### 왜 하나씩만 꺼내는가

- 태스크 사이마다 브라우저가 다른 일을 할 틈을 만들기 위해서다.
- 태스크 하나가 끝나고 다음 태스크를 꺼내기 전 지점에서 브라우저는 렌더링(스타일 계산 → 레이아웃 → 페인트)을 수행할 수 있다. 태스크 10개를 연속으로 붙여서 처리하면 그 사이에 화면이 갱신될 지점이 없다.
- 태스크 소스별 우선순위 재판단도 이 지점에서 일어난다. 한 바퀴 돌 때마다 어느 큐에서 꺼낼지 다시 고르기 때문에, 타이머 태스크가 100개 쌓여 있어도 그 사이에 클릭 이벤트가 들어오면 다음 바퀴에서 클릭을 먼저 처리할 수 있다. 10개를 한꺼번에 처리하면 이 판단 지점이 사라진다.

```js
// 태스크 3개. 각각 사이에 렌더링 기회가 있다
setTimeout(() => {
  box.style.left = "100px";
}, 0);
setTimeout(() => {
  box.style.left = "200px";
}, 0);
setTimeout(() => {
  box.style.left = "300px";
}, 0);
```

- 다만 렌더링은 매 틱마다 반드시 일어나는 게 아니라 화면 주사율(보통 60Hz)에 맞춰 필요할 때만 수행된다. 위 코드가 3프레임에 걸쳐 그려진다는 보장은 없다.

#### 태스크 하나 = 완결 단위

- 이벤트 루프는 태스크를 콜 스택에 push한 뒤, **콜 스택이 다시 완전히 빌 때까지** 아무것도 하지 않는다. 중간에 끼어들지 않는다.
- 그 태스크가 함수를 몇 겹으로 호출하든, 그 안쪽 전부가 끝나야 한 태스크가 끝난 것이다.

```js
setTimeout(() => {
  // 태스크 하나
  a(); // a → b → c 까지 전부 이 태스크에 포함
  console.log("끝");
}, 0);
```

```
콜 스택: [cb]
콜 스택: [cb, a]
콜 스택: [cb, a, b]
콜 스택: [cb, a, b, c]
콜 스택: [cb, a, b]
콜 스택: [cb, a]
콜 스택: [cb]
콜 스택: []          ← 여기서야 이벤트 루프가 다음 태스크를 꺼낸다
```

- 이것이 **run-to-completion**이다. 2번에서 본, 실행 중 코드가 끊기지 않는다는 성질이 이벤트 루프 레벨에서 다시 확인된다.
- 따라서 **한 태스크가 길면 그동안 브라우저 전체가 멈춘다.** 렌더링도 못 하고 클릭도 처리 못 한다. 이걸 블로킹(blocking)이라고 한다.

```js
button.addEventListener("click", () => {
  const end = Date.now() + 3000;
  while (Date.now() < end) {} // 3초짜리 태스크
});
```

- 이 버튼을 누르면 3초 동안 스크롤도 안 되고 다른 버튼도 안 눌린다. 이벤트가 발생하지 않는 게 아니라, 태스크 큐에 쌓이기만 하고 꺼내갈 수 없는 것이다.

#### 확인 문제

```js
console.log("A");

setTimeout(() => {
  console.log("B");
  const end = Date.now() + 2000;
  while (Date.now() < end) {}
  console.log("C");
}, 0);

setTimeout(() => console.log("D"), 100);

console.log("E");
```

1. 출력 순서는?
2. `D`는 100ms 뒤 실행되도록 등록됐다. 실제로는 몇 ms쯤 뒤에 출력되나? 그 이유를 콜 스택과 태스크 큐 상태로 설명해봐.

**답**

1. `A E B C D`
2. 약 2000ms 이상 뒤에 출력된다.

```
0ms    스크립트 태스크 실행. A 출력, 타이머 2개 등록, E 출력
       콜 스택: []  /  태스크 큐: [cb1]      (cb1은 0ms에 만료됨)

0ms    이벤트 루프가 cb1을 콜 스택에 push
       B 출력 → while 진입
       콜 스택: [cb1]  ← 2초 동안 이 상태

100ms  타이머 만료 → cb2가 태스크 큐 진입
       태스크 큐: [cb2]
       하지만 콜 스택이 비어있지 않아 이벤트 루프는 꺼내지 못함

2000ms while 종료, C 출력, cb1 pop
       콜 스택: []

2000ms 이벤트 루프가 cb2를 꺼내 push → D 출력
```

- `setTimeout`의 지연 시간은 "이 시각 이후에 큐에 넣는다"만 보장한다. 큐에서 꺼내지는 시점은 콜 스택이 언제 비느냐에 달려 있다.

### 5. 마이크로태스크 큐

#### 마이크로태스크란

- 태스크 큐와 **별도로 존재하는** 또 하나의 큐다. 여기 들어가는 작업 단위를 **마이크로태스크(microtask)** 라고 한다.
- ECMAScript 명세에서는 **Job**, 그 큐를 `PromiseJobs`라고 부른다. HTML 명세의 microtask queue와 같은 것을 가리킨다. 2번에서 본 "명세가 갈라져 있다"가 여기서 드러난다.

여기 들어가는 것:

| 마이크로태스크를 만드는 것                 | 언제 큐에 들어가는가                 |
| ------------------------------------------ | ------------------------------------ |
| `.then()` / `.catch()` / `.finally()` 콜백 | 프로미스가 settled 되는 시점         |
| `await` 이후 코드                          | await한 프로미스가 settled 되는 시점 |
| `queueMicrotask(fn)`                       | 호출 즉시                            |
| `MutationObserver` 콜백                    | DOM 변경이 감지된 시점               |

- 프로미스 1일차에 "코드 끝 → 엔진이 **대기줄** 꺼내 실행 → B 출력"이라고 적어둔 그 대기줄이 **마이크로태스크 큐**다.
- 태스크 큐와 마찬가지로 FIFO이고, JS에서 접근할 수 없다.

#### 태스크 큐와 분리된 이유

- 프로미스 콜백의 실행 시점을 **일관되게** 만들기 위해서다.

```js
const p1 = Promise.resolve(1); // 이미 fulfilled
const p2 = fetch("/api"); // 나중에 fulfilled

p1.then((v) => console.log("p1", v));
p2.then((v) => console.log("p2", v));
```

- `p1`은 값이 이미 있고 `p2`는 없다. 만약 "값이 있으면 즉시 실행, 없으면 나중에"로 정했다면, `.then` 콜백이 동기적으로 돌 수도 비동기적으로 돌 수도 있게 된다. 호출한 쪽에서는 어느 쪽인지 알 수 없다.

```js
// 만약 즉시 실행되는 경우가 있다면
let data = null;
Promise.resolve(1).then((v) => {
  data = v;
});
console.log(data); // 1일까 null일까? 예측 불가
```

- 그래서 **값이 이미 확정돼 있어도 예외 없이 큐에 넣는다.** 프로미스 1일차의 "콜백은 절대 동기적으로 안 돈다"가 이 규칙이다.
- 그러면 왜 태스크 큐를 쓰지 않았는가. 태스크 큐에 넣으면 다른 태스크(타이머, 클릭 이벤트) 뒤에 줄을 서게 되고, 렌더링도 사이에 끼어든다. 프로미스 체인 중간에 화면이 갱신되면 사용자에게 미완성 상태가 보인다.

```js
fetchUser()
  .then((user) => {
    name = user.name;
  }) // 여기까지 그린 화면이 보이면 안 됨
  .then(() => {
    render();
  });
```

- 마이크로태스크 큐는 **현재 태스크가 끝난 직후, 렌더링과 다음 태스크보다 먼저** 처리된다. 그래서 프로미스 체인 전체가 한 태스크의 연장선처럼 끊김 없이 끝난다.

#### 소진 규칙

- 이벤트 루프 의사코드에 빠져 있던 한 단계가 이것이다.

```
while (브라우저가 살아있는 동안) {
  if (콜 스택이 비어있지 않음) continue;

  task = 태스크 큐에서 하나 꺼냄;
  if (task 있음) {
    task를 콜 스택에 push;
    콜 스택이 다시 빌 때까지 대기;
  }

  // ↓ 추가된 단계
  while (마이크로태스크 큐가 비어있지 않음) {
    m = 마이크로태스크 큐에서 하나 꺼냄;
    m을 콜 스택에 push;
    콜 스택이 다시 빌 때까지 대기;
  }

  렌더링이 필요하면 렌더링;
}
```

- 차이가 결정적이다.

|                   | 한 틱에 처리하는 개수 |
| ----------------- | --------------------- |
| 태스크 큐         | **하나**              |
| 마이크로태스크 큐 | **빌 때까지 전부**    |

- 마이크로태스크를 처리하는 도중 새 마이크로태스크가 추가되면, **그것도 같은 틱에서 처리한다.** 큐가 완전히 빌 때까지 나가지 않는다.

```js
Promise.resolve().then(() => {
  console.log("m1");
  Promise.resolve().then(() => console.log("m2")); // 실행 중에 추가
});
setTimeout(() => console.log("t1"), 0);
console.log("sync");
```

```
sync 출력, 스크립트 태스크 종료
  마이크로태스크 큐: [m1]
  태스크 큐: [t1]

마이크로태스크 소진 시작
  m1 실행 → 'm1' 출력 → m2를 큐에 추가
  마이크로태스크 큐: [m2]        ← 큐가 안 비었으니 계속
  m2 실행 → 'm2' 출력
  마이크로태스크 큐: []          ← 이제 소진 끝

다음 틱: t1 실행 → 't1' 출력
```

- 출력: `sync m1 m2 t1`
- 3번에서 본 태스크와 정반대다. 태스크는 실행 중에 추가하면 다음 틱으로 밀렸는데, 마이크로태스크는 같은 틱에서 처리된다.

#### 기아 상태(starvation)

- "빌 때까지 전부"라는 규칙 때문에, 마이크로태스크가 계속 자기를 재등록하면 큐가 영원히 비지 않는다.

```js
function loop() {
  Promise.resolve().then(loop);
}
loop();

setTimeout(() => console.log("영원히 안 나옴"), 0);
```

- 이벤트 루프는 마이크로태스크 소진 단계에서 빠져나오지 못한다. 결과:
  - 태스크 큐의 `setTimeout` 콜백이 실행되지 않는다
  - 렌더링이 일어나지 않는다 (화면 정지)
  - 클릭 이벤트가 처리되지 않는다
- 콜 스택은 계속 비었다 찼다 반복하므로 스택 오버플로는 나지 않는다. 브라우저는 멈추지만 에러는 없다.
- 같은 코드를 `setTimeout`으로 짜면 이 문제가 없다. 한 틱에 하나만 꺼내므로 사이사이에 다른 작업이 들어갈 자리가 있다.

```js
function loop() {
  setTimeout(loop, 0); // 무한 반복이지만 브라우저는 정상 동작
}
loop();
```

- 이것이 마이크로태스크에 무거운 작업을 넣으면 안 되는 이유다.

#### 확인 문제

```js
console.log("1");

setTimeout(() => {
  console.log("2");
  Promise.resolve().then(() => console.log("3"));
}, 0);

Promise.resolve().then(() => {
  console.log("4");
  setTimeout(() => console.log("5"), 0);
});

Promise.resolve().then(() => console.log("6"));

console.log("7");
```

1. 출력 순서는?
2. `'4'`가 출력된 직후, 태스크 큐와 마이크로태스크 큐에는 각각 무엇이 들어 있나?

**답**

1. `1 7 4 6 2 3 5`

```
[스크립트 태스크]
  '1' 출력
  setTimeout(A, 0) 등록 → 만료 → 태스크 큐: [A]
  then(B) 등록 → p가 이미 fulfilled → 마이크로태스크 큐: [B]
  then(C) 등록 → 마이크로태스크 큐: [B, C]
  '7' 출력
  콜 스택: []

[마이크로태스크 소진]
  B 실행 → '4' 출력 → setTimeout(D, 0) 등록 → 만료 → 태스크 큐: [A, D]
  C 실행 → '6' 출력
  마이크로태스크 큐: []

[다음 틱 — 태스크 하나]
  A 실행 → '2' 출력 → then(E) → 마이크로태스크 큐: [E]
  A 종료, 콜 스택: []

[마이크로태스크 소진]     ← 다음 태스크(D)보다 먼저
  E 실행 → '3' 출력

[다음 틱]
  D 실행 → '5' 출력
```

2. `'4'` 직후 상태:
   - 태스크 큐: `[A, D]` — A는 대기 중, D는 방금 추가됨
   - 마이크로태스크 큐: `[C]` — B는 꺼내져서 콜 스택에 있으므로 큐에 없음

- `'5'`가 `'3'`보다 뒤인 것이 핵심이다. D는 A보다 먼저 큐에 들어갔지만 A가 먼저 실행됐고(FIFO), A가 만든 마이크로태스크 E가 D보다 먼저 처리됐다.

### 6. 종합 실행 순서

#### 전체 순서 규칙

세 줄로 압축된다.

```
1. 동기 코드를 끝까지 실행한다 (콜 스택이 빌 때까지)
2. 마이크로태스크 큐를 빌 때까지 전부 실행한다
3. 태스크 큐에서 하나 꺼내 실행한다 → 2번으로 돌아간다
```

- 실행 중 새로 등록되는 것은 각자의 큐 맨 뒤로 간다.
- 마이크로태스크는 같은 사이클에서 소진되고, 태스크는 다음 사이클로 밀린다.

|              | 콜 스택             | 태스크 큐                          | 마이크로태스크 큐                       |
| ------------ | ------------------- | ---------------------------------- | --------------------------------------- |
| 담는 것      | 실행 컨텍스트       | 함수 참조                          | 함수 참조                               |
| 순서         | LIFO                | FIFO                               | FIFO                                    |
| 대표 항목    | 실행 중인 모든 함수 | `setTimeout`, DOM 이벤트, 네트워크 | `.then`, `await` 이후, `queueMicrotask` |
| 한 틱 처리량 | —                   | 1개                                | 전부                                    |
| 소속 명세    | ECMAScript          | HTML                               | ECMAScript(Job) / HTML                  |

- 자주 나오는 오해 정정:
  - ❌ "`setTimeout(f, 0)`은 즉시 실행" → 최소 지연이고, 콜 스택이 빈 뒤에 실행된다
  - ❌ "프로미스는 항상 `setTimeout`보다 빠르다" → 큐가 달라서 그런 것이지, 등록 시점과 무관하게 항상 빠른 건 아니다. 이미 실행 중인 태스크는 먼저 끝난다
  - ❌ "`.then`은 값이 있으면 바로 실행" → 예외 없이 마이크로태스크 큐를 거친다
  - ❌ "JS는 싱글 스레드라 동시 처리를 못 한다" → JS 실행 스레드가 하나일 뿐, 타이머·네트워크는 브라우저의 다른 스레드가 처리한다

#### 확인 문제

```js
console.log("start");

setTimeout(() => console.log("t1"), 0);

new Promise((resolve) => {
  console.log("p1");
  resolve();
  console.log("p2");
})
  .then(() => {
    console.log("p3");
    return Promise.resolve();
  })
  .then(() => {
    console.log("p4");
  });

queueMicrotask(() => console.log("q1"));

setTimeout(() => {
  console.log("t2");
  queueMicrotask(() => console.log("q2"));
}, 0);

console.log("end");
```

1. 출력 순서는?

**답**

1. `start p1 p2 end p3 q1 p4 t1 t2 q2`

```
[스크립트 태스크]
  'start'
  setTimeout(t1) → 태스크 큐: [t1]
  executor 즉시 실행 → 'p1' → resolve() → 'p2'
    (executor는 동기다. 프로미스 1일차 참조)
  .then(p3) 등록 → 이미 fulfilled → 마이크로태스크 큐: [p3콜백]
    (두 번째 .then은 아직 pending인 프로미스에 등록되므로 큐에 안 들어감)
  queueMicrotask(q1) → 마이크로태스크 큐: [p3콜백, q1]
  setTimeout(t2) → 태스크 큐: [t1, t2]
  'end'

[마이크로태스크 소진]
  p3콜백 실행 → 'p3' → 프로미스를 반환
  q1 실행 → 'q1'
  (반환된 프로미스가 풀리면서 p4콜백이 큐에 추가됨)
  p4콜백 실행 → 'p4'
  마이크로태스크 큐: []

[틱 — 태스크 1개]
  t1 → 't1'
[마이크로태스크 소진] 없음

[틱 — 태스크 1개]
  t2 → 't2' → queueMicrotask(q2) → 마이크로태스크 큐: [q2]
[마이크로태스크 소진]
  q2 → 'q2'
```

- `q1`이 `p4`보다 먼저인 것이 함정이다. `.then` 콜백이 프로미스를 반환하면 그 프로미스를 풀기 위해 추가 마이크로태스크가 소비되므로, 다음 `.then`이 한 틱 뒤로 밀린다. 이 부분은 프로미스 2·3일차에서 정확히 다룬다.

## 틀렸거나 부족했던 부분

- `while`, `for`, `if`는 실행 컨텍스트를 만들지 않는다. 실행 컨텍스트는 **스크립트 평가**와 **함수 호출** 시점에만 생성된다.
- 콜 스택 상태를 적을 때 **전역 실행 컨텍스트를 빠뜨렸다.** 스크립트가 끝나기 전까지 스택 바닥에는 항상 전역이 있다.
- "태스크 큐에서 실행된다"는 표현은 틀렸다. 실행되는 장소는 언제나 **콜 스택**이다. 큐는 대기 장소일 뿐이고, 꺼내져서 콜 스택에 push된 뒤에 실행된다.
- `setTimeout(cb, 0)` 호출 시 콜 스택에 올라가는 것은 `setTimeout` 자체이지 `cb`가 아니다. `cb`는 인자로 넘어간 함수 객체이고, 이 시점에 실행 컨텍스트가 없다.
- 태스크 실행 중 새로 등록한 태스크는 **큐 맨 뒤**로 간다. 지금 것 바로 다음이 아니다.
- 동기 코드가 비동기 콜백보다 항상 먼저다. 등록 순서가 아니라 큐 종류가 순서를 결정한다.

## 면접용 1분 답변

```
"이벤트 루프에 대해 설명해주세요."

자바스크립트 엔진은 콜 스택과 힙만 가지고 있고, 실행 스레드는 하나입니다. 함수가 호출되면 실행 컨텍스트가 콜 스택에 push되고, 반환되면 pop됩니다. 스택 최상단이 현재 실행 중인 코드이고, 스택에 컨텍스트가 남아 있는 동안에는 다른 코드가 끼어들 수 없습니다.

그런데 setTimeout이나 fetch, DOM 이벤트는 ECMAScript 명세에 없습니다. 이것들은 브라우저나 Node 같은 호스트 환경이 제공하고, 실제 대기 처리도 호스트 환경의 별도 스레드에서 이루어집니다. setTimeout을 호출하면 브라우저 타이머 모듈에 콜백과 만료 시각을 등록하고 즉시 반환되기 때문에, 콜 스택은 바로 비워집니다.

타이머가 만료되면 브라우저는 콜백을 콜 스택에 직접 올리지 않고 태스크 큐에 넣습니다. 실행 중인 코드 중간에 끼어들면 지역 변수 상태가 깨질 수 있기 때문입니다. 이벤트 루프는 콜 스택이 비었는지 확인하고, 비어 있을 때만 큐에서 하나를 꺼내 콜 스택에 올립니다. 한 번 올라간 코드는 끝까지 실행되며 중단되지 않고, 이를 run-to-completion이라고 합니다.

큐는 두 종류입니다. setTimeout이나 DOM 이벤트 콜백이 들어가는 태스크 큐, 그리고 프로미스 콜백과 queueMicrotask, await 이후 코드가 들어가는 마이크로태스크 큐입니다. 둘의 처리 방식이 다릅니다. 태스크 큐는 한 틱에 하나만 꺼내지만, 마이크로태스크 큐는 완전히 빌 때까지 전부 처리합니다. 처리 도중 추가된 마이크로태스크까지 같은 틱에서 소진합니다.

따라서 실행 순서는 동기 코드, 마이크로태스크 전부, 태스크 하나 순이고 이것이 반복됩니다. 프로미스 콜백이 setTimeout보다 먼저 실행되는 것은 프로미스가 빨라서가 아니라 소속된 큐가 다르기 때문입니다. 프로미스는 값이 이미 확정되어 있어도 예외 없이 마이크로태스크 큐를 거치는데, 콜백 실행 시점을 항상 비동기로 통일해 순서를 예측 가능하게 만들기 위해서입니다.

주의할 점은 두 가지입니다. 첫째, 태스크 하나가 오래 걸리면 그동안 렌더링과 이벤트 처리가 전부 멈춥니다. 둘째, 마이크로태스크가 자기 자신을 계속 등록하면 큐가 비지 않아 태스크와 렌더링이 영원히 실행되지 못하는 기아 상태가 발생합니다.
```

## 한 줄 요약

- 이벤트 루프는 콜 스택이 빈 것을 확인하고 마이크로태스크 큐를 전부 소진한 뒤 태스크 큐에서 하나를 꺼내 콜 스택에 올리는 반복문이며, 이 순서가 자바스크립트의 모든 비동기 실행 순서를 결정한다.
