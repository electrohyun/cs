# JavaScript 프로미스 (Promise) — 1일차: 개념·상태·체이닝

## 잘 알고 있는지

## 내가 아는 만큼 설명

- Promise = 약속
- 비동기 행위의 결과를 담기 위한 객체 정도로만 알고 있다.

## 답을 본 뒤 알게 된 것

### 프로미스란

- 비동기 작업의 최종 결과를 담을 그릇 객체
- 지금은 값이 없지만, 나중에 성공값이나 실패 이유 둘 중 하나가 담기기로 약속된 객체.
- 핵심은 함수 호출 즉시 리턴!
- 그래서 그 리턴값을 변수에 넣거나, 배열에 모으거나 할 수 있다. 콜백은 이게 안됐다!

### 프로미스 상태 3가지

1. pending - 아직 결과 안 나옴. 초기 상태
2. fulfilled - 성공, 값이 확정됨
3. rejected - 실패. 이유가 확정됨

- 특징으로는 peding에서 fuilfilled나 rejected로 딱 한 번만 이동함
- 이 확정 상태를 `settled` 라고 부름

### 예제 코드

```js
const p = new Promise((resolve, reject) => {
  resolve("첫번째");
  resolve("두번째"); // 무시됨
  reject("에러"); // 무시됨
});
```

- "첫 resolve에서 상태가 굳고, 나머지는 그냥 씹혀. 아까 콜백의 "두 번 불리면 두 번 실행됨" 문제가 여기서 구조적으로 막히는 거야. 그리고 이 불변성 덕에 결과가 이미 나온 뒤에 .then을 붙여도 값을 받을 수 있어. 이벤트 리스너는 이미 지나간 이벤트를 못 받잖아. 프로미스는 상태가 박제돼 있으니까 언제 붙어도 그 값을 준다." by Claude

### resolve / reject

```js
new Promise((resolve, reject) => {
  // 이 안은 즉시 실행됨
});
```

- new Promise에 넘기는 함수를 **executor(실행자)**라고 부른다.
- 그리고 이 executor는 자바스크립트 엔진이 즉시, 동기적으로 호출한다.
- new Promise 만나는 순간 바로 실행

```
resolve(value) → pending을 fulfilled로 확정, 값은 value
reject(reason) → pending을 rejected로 확정, 이유는 reason
```

- pending에서 한 번만 이동한다고 말 했을때, 그 이동시키는 녀석들이 바로 이것
- 그래서 첫 번째 호출만 가능
- 이름은 그냥 관례이다.

```js
const delay = (ms) => new Promise((resolve) => setTimeout(resolve, ms));
```

- "하나만 더 — 실무에서 new Promise를 직접 쓸 일은 생각보다 적어. fetch나 라이브러리가 이미 프로미스를 만들어서 주니까. 직접 쓰는 대표 케이스는 콜백 기반 옛날 API를 프로미스로 감쌀 때야." by Claude

### then, catch, finally

- 프로미스에 딱 붙는 메서드 3가지!
- fetch를 사용해봤으면 봤을 것
- all이나 race 같은 Promise. 으로 사용하는 정적 메서드랑은 성격이 다르다(4일차에 하기)

#### then

```js
p.then(
  (value) => {}, // fulfilled면 실행, value = resolve로 넘긴 값
  (reason) => {}, // rejected면 실행, reason = reject로 넘긴 값
);
```

- 확정됐을 때 실행할 함수를 등록하는 메서드. 인자를 두 개 받아.
- value => {}, // fulfilled면 실행, value = resolve로 넘긴 값
- reason => {} // rejected면 실행, reason = reject로 넘긴 값

- 두 개 중 상태에 맞는 하나만 실행된다. 둘 다 실행되는 일은 없다.
- 등록이지 실행이 아니다. 부르는 시점에 콜백이 도는 게 아니라 예약하는 것
- 콜백은 절대 동기적으로 안 돈다. 이미 fulfilled인 프로미스에 붙여도, 현재 동기 코드가 전부 끝난 뒤에 실행된다. (3일차에 하기)

##### then은 새 프로미스를 반환한다

```js
const p2 = p1.then((v) => v * 2);
// p2는 p1이 아니라, then이 새로 만든 프로미스
```

- .then()의 리턴값은 또 프로미스임
- .then()을 붙일 수 있음. = 이게 체이닝.
- 프로토타입 할 때 본 "메서드가 값을 반환해서 이어붙는 구조" 그대로.

```js
Promise.resolve(1)
  .then((v) => v + 1) // 2를 담은 새 프로미스
  .then((v) => v * 10) // 20을 담은 새 프로미스
  .then((v) => console.log(v)); // 20
```

- 새 프로미스에는 콜백이 리턴한 값이 붙는다.
- 리턴을 안 하면? undefined가 담긴다.
- 함수 기본 동작 그대로!
- 체인 중간에서 값이 갑자기 undefined로 바뀌는 사고는 대부분 여기서 남

- "특별 규칙 하나 — 콜백이 프로미스를 리턴하면, 그 프로미스가 확정될 때까지 기다렸다가 안의 값을 꺼내서 넘겨. 프로미스가 프로미스로 이중 포장되지 않아. 비동기 작업을 평평하게 이어붙일 수 있는 이유가 이거야." by Claude

#### catch

```js
p.catch(fn);
// === p.then(undefined, fn)

step1()
  .then(step2)
  .then(step3)
  .catch(err => ...);  // 1,2,3 어디서 터져도 여기
```

- then의 두 번째 인자만 쓰는 것과 똑같다.
- 진짜로 내부에서 그렇게 정의되어 있다고 함.
- 새로 생긴 기능이 아니라 이름을 붙여준 것.
- 체인 맨 끝에 두면 어디에 있던 간에 여기로 흘러 들어옴.

- "콜백 시절에 단계마다 if (err) 쓰던 게 하나로 합쳐지는 지점. 자세한 전파 규칙은 4일차." by Claude

#### finally

```js
setLoading(true);
fetchData()
  .then(...)
  .catch(...)
  .finally(() => setLoading(false));
```

- 성공이든 실패든 무조건 실행. 인자를 안 받음. 성공값도 에러도 안 넘어옴.
- 로딩 스피너 끄기, 연결 닫기처럼 결과와 무관하게 치워야 하는 일에 씀.
- 그리고 값을 통과시킴. finally를 거쳐도 뒤의 then은 값을 그대로 받음.

#### 덤) Promise.resolve / Promise.reject

```js
Promise.resolve(42); // 42로 fulfilled된 프로미스
Promise.reject(new Error("x")); // rejected된 프로미스
```

- 이미 확정된 프로미스를 즉석에서 만드는 단축키
- new Promise(res => res(42)) 쓰기 귀찮을 때, 그리고 값이 프로미스일 수도 아닐 수도 있을 때 강제로 프로미스로 맞출 때 사용함.

### 확인 문제

```js
const p = new Promise((resolve) => {
  console.log("A");
  resolve("B");
});
console.log("C");
p.then((v) => console.log(v));
```

- 실행 결과가 뭘까?
- 답은 A -> C -> B.

1. Promise는 함수 호출 즉시 리턴이다. 콘솔 로그 A 출력.
2. B는 실제 코드면 실제 결과값이 저장되는 것임. 이 프로미스의 결과값은 B다, 그리고 지금부터 fulfilled다.
3. 콘솔 로그 C가 출력된다.
4. p.then(...)은 콜백을 등록만 한다. 실행은 동기 코드가 전부 끝난 뒤.

### 헷갈려요

- then 안의 익명 함수가 헷갈린다! 풀어서 보자.

```js
function executor(resolve) {
  console.log("A");
  resolve("B");
}

function printValue(v) {
  console.log(v);
}

const p = new Promise(executor);
console.log("C");
p.then(printValue);
```

- 프로미스 객체는 안에 이런걸 들고 있다.

```js
p = {
  상태: "pending",
  값: undefined,
  콜백목록: [], // ← 등록된 함수들이 여기 쌓임
};
```

- resolve("B")는 값 칸과 상태 칸을 채움.

```js
상태: "fulfilled";
값: "B";
```

- p.then(printValue)는 콜백목록에 함수를 넣음.

```js
콜백목록: [printValue];
```

- 둘 다 채워지면 엔진이 콜백목록을 돌면서 값 칸을 인자로 넣어 실행.

```js
printValue("B");
```

- then의 함수가 resolve로 흘러가는 게 아니라, 둘 다 프로미스 객체라는 같은 상자에 각자 자기 것을 넣는 구조.

### 순서가 반대여도 괜찮아요

- resolve가 먼저 (지금 예제) — 값이 이미 있는데 콜백목록이 비어 있어. then이 등록되는 순간 "어, 값 이미 있네" 하고 바로 실행 예약.

- then이 먼저 (실제 비동기) — 콜백은 등록됐는데 값이 아직 없어. 목록에 대기시켜 두고, 나중에 resolve가 불리면 그때 꺼내서 실행.

```js
const p = new Promise((resolve) => {
  setTimeout(() => resolve("B"), 1000);
});
p.then(printValue); // 등록. 1초 대기
```

- "resolve도 마찬가지로 엔진이 만들어서 executor에 꽂아준 함수야. 정체는 "값 칸과 상태 칸을 채우고 콜백목록을 실행하는 함수". 네가 정의하지 않았을 뿐이지 printValue랑 똑같은 평범한 함수고, 프로미스 객체 내부를 건드릴 권한만 갖고 있어." by Claude

### 정리

```js
const p = new Promise((resolve) => {
  console.log("A");
  resolve("B");
});
console.log("C");
p.then((v) => console.log(v));
```

- executor 즉시 실행 → A 출력
- resolve("B") → 값 "B", 상태 fulfilled 확정. 콜백목록 비어서 세울 게 없음
- C 출력
- then이 콜백 등록 → 값이 이미 있으니 대기줄에 세움
- 코드 끝 → 엔진이 대기줄 꺼내 실행 → B 출력

## 틀렸거나 부족했던 부분

- 프로미스 3가지 상태
- 리졸브와 리젝트란
- then의 진짜 역할

## 면접용 1분 답변

```
"프로미스가 뭔가요?"

프로미스란 비동기 작업의 최종 결과를 담을 그릇 객체입니다. 함수를 호출하면 결과를 기다렸다가 주는 게 아니라, "나중에 여기 값이 담길 것"이라는 객체를 즉시 반환합니다.

프로미스 이전에는 콜백 함수를 인자로 넘겨서 결과를 처리했는데, 여기엔 세 가지 문제가 있었습니다. 첫째로 제어권이 넘어갑니다. 내 콜백이 언제, 몇 번 불릴지를 호출당한 함수가 결정합니다. 둘째로 결과가 손에 남지 않습니다. 리턴값이 없어서 변수에 담거나 다른 곳에 넘길 수 없습니다. 셋째로 에러 처리가 흩어집니다. 단계마다 따로 처리해야 하고, 콜 스택이 이미 넘어가서 바깥 try-catch로도 잡히지 않습니다. 흔히 말하는 콜백 지옥은 들여쓰기가 아니라 이 문제들이 본질입니다.

프로미스는 세 가지 상태를 갖습니다. 아직 결과가 없는 pending, 성공해서 값이 확정된 fulfilled, 실패해서 이유가 확정된 rejected입니다. pending에서 나머지 두 상태 중 하나로의 전이는 단 한 번만 일어나고 비가역적입니다. 확정된 상태를 settled라고 부릅니다.

이 상태 변화는 new Promise에 넘기는 executor 함수 안에서 일으킵니다. executor는 즉시 동기적으로 실행되고, 인자로 받은 resolve나 reject를 호출해 값과 상태를 확정합니다. 이때 이미 확정된 프로미스에 다시 resolve를 호출해도 무시되기 때문에, 콜백에서 문제가 됐던 중복 실행이 구조적으로 차단됩니다.

확정된 값은 then으로 꺼냅니다. then은 실행할 콜백을 등록하는 메서드이고, 콜백은 현재 실행 중인 동기 코드가 모두 끝난 뒤에 실행됩니다. 값이 이미 확정되어 있어도 예외 없이 그렇습니다. 실행 시점을 일관되게 만들어 순서를 예측 가능하게 하기 위해서입니다.

또한 then은 새로운 프로미스를 반환하기 때문에 연속해서 이어붙일 수 있습니다. 콜백이 반환한 값이 다음 then으로 전달되고, 콜백이 프로미스를 반환한 경우에는 그것이 확정될 때까지 기다렸다가 내부 값을 꺼내 전달합니다. 이 덕분에 중첩되던 비동기 작업이 평평한 체인으로 펴집니다. 에러는 체인을 타고 흐르므로 마지막에 catch 하나만 두면 어느 단계에서 발생하든 한곳에서 처리할 수 있고, 성공·실패와 무관하게 실행할 정리 작업은 finally에 둡니다.

마지막으로 프로미스는 값처럼 다룰 수 있는 객체입니다. 변수에 담고, 배열에 모으고, 인자로 넘기고, 반환할 수 있습니다. Promise.all처럼 여러 비동기 작업을 조합하는 도구가 가능한 것도 이 성질 덕분입니다. 콜백은 반환값이 없어 애초에 조합의 대상이 될 수 없었습니다.
```

## 한 줄 요약

- 프로미스는 비동기 작업의 결과와 상태를 담는 객체이고, resolve가 값을 확정하면 then으로 등록한 콜백이 동기 코드 종료 후 그 값을 받아 실행된다.
